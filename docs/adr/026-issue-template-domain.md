# ADR-026: Issue-template domain — generator scope, schema-derived enums, and ADR-content alignment contract

**Status:** Accepted
**Date:** 2026-05-19
**Authors:** Architect agent, with maintainer review

---

## Context

The repository ships GitHub issue templates under `.github/ISSUE_TEMPLATE/`. A subset of those templates is generated by [`scripts/generate_issue_templates.py`](../../scripts/generate_issue_templates.py) from sources under [`scripts/TEMPLATES/`](../../scripts/TEMPLATES/), wired through the pre-commit framework by the `regenerate-issue-templates` and `validate-issue-templates` hooks ([ADR-005](005-pre-commit-framework.md)). The generator expands schema-derived placeholders (`{{RISK_CATEGORIES}}`, `{{CONTROL_CATEGORIES}}`, `{{COMPONENT_CATEGORIES}}`, `{{PERSONAS}}`, `{{PERSONAS_FOR_RISKS}}`, `{{LIFECYCLE_STAGE}}`, `{{IMPACT_TYPE}}`, `{{ACTOR_ACCESS}}`) by reading the JSON schemas under `risk-map/schemas/` and filtering by `deprecated: true` against `risk-map/yaml/personas.yaml`. The pre-commit hook config explicitly documents the dual-trigger pattern: any edit under `scripts/TEMPLATES/`, `risk-map/schemas/`, or `risk-map/yaml/frameworks.yaml` regenerates the full set and `git add`s `.github/ISSUE_TEMPLATE/` (Mode B auto-stage).

Three forces motivate this ADR.

**Force 1 — silent generator-coverage gap.** Eight templates exist for the four content-entity types (`new_{risk,control,component,persona}` and `update_{risk,control,component,persona}`), but `scripts/TEMPLATES/` carries only five source files. `IssueTemplateGenerator.get_available_templates()` does `glob("*.template.yml")` over `scripts/TEMPLATES/`, so the generator empirically reports `Success: 5/5`. The four un-sourced templates (`new_persona.yml`, `update_persona.yml`, `update_component.yml`, plus the entity-type-neutral `infrastructure.yml`) are hand-edits. The defensive `entity_mappings` table in `generator.py` lists `new_persona`, `update_persona`, and `update_component` — those entries are dead-code allowances because the glob never reaches them. Net effect: a closed-enum schema change (a new persona ID, a new component ID, a new lifecycle stage) propagates to risk and control templates but not to persona or `update_component` templates. The orphans are guaranteed to drift on every schema enum change. The repository's documented rule "edit `scripts/TEMPLATES/*.template.yml` sources first; the generated outputs are wiped on regeneration" is correct for the five sources that exist but does not address the four orphans.

**Force 2 — content drift against ADRs 016/017/019/020/021/022.** Since [ADR-016](016-reference-strategy.md) (`externalReferences[]` and the `{{ref:identifier}}` sentinel grammar), [ADR-017](017-yaml-prose-authoring-subset.md) (prose subset; raw HTML rejected; inline URLs rejected), [ADR-019](019-risks-schema.md) D6 (`relevantQuestions` retirement), [ADR-021](021-personas-and-self-assessment-schema.md) (`responsibilities`, `mappings`, `identificationQuestions`), and [ADR-022](022-supporting-schemas.md) D5b (canonical per-framework mapping-ID regex patterns) landed, the issue templates have not been updated to match. Concrete drift, by template:

| Drift | Where | ADR violated | Severity |
|---|---|---|---|
| Raw `<a href>` example in `examples` placeholder; "Can include HTML links" helper text | `new_risk.template.yml`, `update_risk.template.yml` | [ADR-017](017-yaml-prose-authoring-subset.md) D1/D2 | FAIL — contributors submit HTML that the prose lint will reject |
| `relevant-questions` field solicited; "Update relevant questions" change-type option | `new_risk.template.yml`, `update_risk.template.yml` | [ADR-019](019-risks-schema.md) D6 | FAIL post-conformance-sweep — submissions will fail schema validation |
| No `externalReferences[]` solicitation | all eight entity templates | [ADR-016](016-reference-strategy.md) D3 | FAIL on risks (≈131 live instances), WARN on controls/components/personas |
| No guidance on the `{{<entity-id>}}` / `{{ref:identifier}}` sentinel grammar | all eight entity templates | [ADR-016](016-reference-strategy.md) D2, [ADR-017](017-yaml-prose-authoring-subset.md) D1 | FAIL — contributors submit bare camelCase IDs and inline HTML anchors |
| STRIDE placeholder uses kebab-case (`information-disclosure`); NIST AI RMF placeholder uses `GV-6.2` form; OWASP placeholder uses `LLM01` (no year) | risk/control templates | [ADR-022](022-supporting-schemas.md) D5b regex commitments (PascalCase STRIDE, `(GOVERN\|MAP\|MEASURE\|MANAGE)-…`, `LLM\d{2}:\d{4}`) | WARN — placeholders teach the wrong canonical form |
| `component-subcategory` declared as free-form `type: input` | `new_component.template.yml` | components schema closes the subcategory `id` to seven values | FAIL — contributors submit free-form text that fails schema validation |
| No `mappings` field solicitation on components | `new_component.template.yml`, `update_component.yml` (orphan) | [ADR-018](018-components-schema.md) D6, [ADR-022](022-supporting-schemas.md) D5c | WARN |
| No `responsibilities`, `mappings`, or `externalReferences` solicitation on personas | `new_persona.yml` (orphan), `update_persona.yml` (orphan) | [ADR-021](021-personas-and-self-assessment-schema.md) schema | FAIL |

The drift is not a one-time backlog. Without a contract that templates teach the conventions established by the ADRs they solicit fields from, the same drift will recur at every future ADR that touches a content schema or an authoring convention.

**Force 3 — no policy on schema-derived enum closure.** The generator's `PLACEHOLDER_MAPPINGS` table covers seven schema enums today. Two new enum surfaces are demonstrably under-covered: the components-subcategory `id` (free-form `input` in the template; closed enum in the schema) and any future closed enum a schema gains. The repository has no documented rule for "every closed-enum schema field that a template solicits MUST use a placeholder." Without that rule, the next schema enum addition will face the same gap discovery the audit surfaced for subcategory.

The current state was effectively frozen at PR #209 (April 2026, "backports source templates to current spec") and an audit on 2026-05-19. The maintainer's framing is that this is acknowledged tooling tech-debt and the decision is not "fix the audit findings" but "what is the policy for the issue-template domain going forward."

This ADR sits adjacent to:

- [ADR-005](005-pre-commit-framework.md) — owns the `regenerate-issue-templates` and `validate-issue-templates` hooks and the Mode B auto-stage pattern. The ADR-005 Addendum 2026-05-08 carves the generator out of the trigger-vs-read-set invariant explicitly. This ADR does not change ADR-005's hook design; it adds a contract that the source layer feeding those hooks must satisfy.
- [ADR-016](016-reference-strategy.md) — owns the `externalReferences[]` schema and the `{{ref:identifier}}` sentinel grammar. Templates teach this; they do not author it.
- [ADR-017](017-yaml-prose-authoring-subset.md) — owns the prose authoring subset (markdown subset, no raw HTML, no inline URLs). Templates' helper text must agree with what the prose lint enforces.
- [ADRs 018/019/020/021](018-components-schema.md) — own the four content schemas. Templates solicit fields from these.
- [ADR-022](022-supporting-schemas.md) D5b — owns the per-framework mapping-ID regex patterns. Template placeholders teach the canonical form.

## Decision

The issue-template domain is a coherent architectural surface with three layered contracts: a **source-coverage contract** (which templates regenerate from source), a **schema-derived-enum contract** (which schema fields appear as placeholders), and an **ADR-content alignment contract** (which authoring conventions the templates teach).

### D1. Surface scope

The issue-template domain consists of three file groups, with explicit roles:

| Group | Path | Role | Edit rules |
|---|---|---|---|
| Sources | `scripts/TEMPLATES/*.template.yml` | Hand-authored. The canonical surface for any template that solicits a content entity. | Edit here; the generator wipes generated outputs on regeneration. |
| Generated outputs | `.github/ISSUE_TEMPLATE/{new,update}_{risk,control,component,persona}.yml` | Machine-produced from sources. | Do not hand-edit. The pre-commit hook regenerates and `git add`s on every source/schema/frameworks edit. |
| Non-content templates | `.github/ISSUE_TEMPLATE/infrastructure.yml`, `.github/ISSUE_TEMPLATE/config.yml` | Hand-edited. Not tied to a content schema; `entity_type = None` per `generator._get_entity_type`. | Edit directly. Excluded from this ADR's coverage contract by construction. |

This ADR governs the first two groups. The third group is out of scope: `infrastructure.yml` is a generic engineering-issue intake and `config.yml` is GitHub's repo-level issue chooser config. Neither solicits a content-entity field; neither will ever have a `scripts/TEMPLATES/` source.

### D2. Source-coverage contract — eight entity templates, eight sources

Every content-entity-type that has a content schema under `risk-map/schemas/` carries **both** a `new_*` and an `update_*` source-tracked template. Four entity types (`risks`, `controls`, `components`, `personas`) × two operations (`new`, `update`) = eight source files at `scripts/TEMPLATES/*.template.yml`. The current state (five sources / four orphans) is a transitional artifact and is closed by this ADR.

Concretely, three new source files are added by the implementation phase that follows this ADR:

- `scripts/TEMPLATES/new_persona.template.yml`
- `scripts/TEMPLATES/update_persona.template.yml`
- `scripts/TEMPLATES/update_component.template.yml`

After the backfill, `IssueTemplateGenerator.get_available_templates()` returns eight names, `python scripts/generate_issue_templates.py --dry-run` reports `Success: 8/8`, and the `entity_mappings` table entries that were previously dead-code allowances become live mappings.

The defensive `entity_mappings` table stays in `generator.py` as it is. Its presence ensures that any future source-file addition for an in-scope entity type is a pure data change in `scripts/TEMPLATES/`, not a coupled `(source-file, generator-code)` change. New content-entity schemas (if any are ever added) require a coordinated `entity_mappings` table update; that coupling is acceptable because adding a content entity type is itself an architecturally-significant change that would warrant its own ADR.

### D3. Schema-derived-enum contract — closed-enum closure rule

**Rule.** Every closed-enum schema field that a template solicits MUST be rendered via a placeholder fed from `TemplateRenderer.PLACEHOLDER_MAPPINGS`. Free-form `type: input` is rejected for fields whose schema declares a closed enum.

**Mechanism.** `PLACEHOLDER_MAPPINGS` is the single registry of `(placeholder-name, schema-path, field-type)` triples. Adding a new closed-enum solicitation to any template requires adding the corresponding placeholder entry to `PLACEHOLDER_MAPPINGS` first. The order is enforced by the generator: an unrecognized placeholder in a source file raises at render time.

**Immediate scope.** The components-subcategory `id` enum (seven values: `componentsModelTraining`, `componentsData`, `componentsAgent`, `componentsOrchestration`, `componentsModelDeployment`, `componentsModelCore`, `componentsApplicationCore`) gains a `{{COMPONENT_SUBCATEGORIES}}` placeholder and the component templates' free-form subcategory `input` becomes a dropdown.

**Out-of-scope.** Open-enum or pattern-constrained fields (`externalReferences[].id`, `mappings.<framework>` per-entry IDs, prose fields) are not subject to the closure rule; they remain `textarea` with helper text that cites the canonical form and the ADR that owns the pattern. Closed-enum fields whose values are not currently solicited by any template (e.g., `externalReferences[].type`) become subject to this rule only if a future template adds a solicitation for them.

### D4. ADR-content alignment contract

Templates are contributor-facing teaching surfaces. They must teach the authoring conventions established by the ADRs governing the schemas they solicit fields from. This contract has three components.

#### D4a. Per-template ADR-citation manifest

Each source template carries, in a YAML comment block at the top of the file, the list of ADRs whose conventions it teaches. The comment is informational (not parsed) and exists so that a reviewer of an ADR amendment can mechanically discover which templates a given ADR change implicates. Example shape:

```yaml
# ADR-content alignment manifest:
# - ADR-014 (YAML content security posture)
# - ADR-016 (externalReferences[] + {{ref:…}} sentinel grammar)
# - ADR-017 (prose subset — no raw HTML, no inline URLs)
# - ADR-019 (risks schema — no relevantQuestions)
# - ADR-022 D5b (framework-mapping regex — STRIDE PascalCase, NIST GOVERN-/MAP-/MEASURE-/MANAGE-, OWASP LLMNN:YYYY)
```

#### D4b. Helper-text canonicality

Source templates teach the **canonical** form of any convention they reference, never a non-canonical form, even where validators accept loose input through fall-through paths. Concretely:

- STRIDE placeholder values use PascalCase (`InformationDisclosure`), matching ADR-022 D5b's `^(Spoofing\|Tampering\|Repudiation\|InformationDisclosure\|DenialOfService\|ElevationOfPrivilege)$` pattern, not kebab-case.
- NIST AI RMF placeholder values use the full function-prefix form (`GOVERN-1.1`, `MAP-2.3`, `MEASURE-1.1.1`, `MANAGE-1.1`), matching ADR-022 D5b's `^(GOVERN\|MAP\|MEASURE\|MANAGE)-\d+(\.\d+)*$` pattern, not the legacy `GV-/MP-/MS-/MG-` abbreviations.
- OWASP Top 10 for LLMs placeholder values use the versioned form (`LLM01:2025`), matching ADR-022 D5b's `^LLM\d{2}:\d{4}$` pattern, not the bare `LLM01` form.
- Prose-field helper text and example placeholder bodies use markdown-subset tokens only (`**bold**`, `*italic*`, `_italic_`, `{{<entity-id>}}` sentinels, `{{ref:identifier}}` sentinels) per ADR-017 D1. Raw HTML examples and inline URLs in helper text are rejected.
- Prose helper text for any entity type that has `externalReferences[]` in its schema (risks, controls, components, personas, per ADR-016 D3) describes the structured-citation shape (`type`, `id`, `title`, `url`) and the `{{ref:identifier}}` sentinel resolution rule.

#### D4c. Style-guide cross-references are aware of D4b

The contributing-guide pages linked from template helper text (`risk-map/docs/contributing/{submission-readiness-guide,common-review-findings,risk-titles-style-guide,control-titles-style-guide,component-titles-style-guide,framework-mappings-style-guide,identification-questions-style-guide}.md`) carry the same canonical-form rules as the templates that link to them. Templates do not invent guidance the linked guide contradicts. Where a style guide is out of date relative to an ADR, the ADR is the source of truth and the guide is updated to match before a template change relies on it.

This is a documentation discipline, not a mechanical enforcement; the mechanical layer is the prose lint (ADR-017 D4) and the schema validators. The contract here is that the documentation chain (template → linked guide → ADR) presents one consistent story.

### D5. Generator coupling and drift-prevention

The pre-commit hook configuration ([ADR-005](005-pre-commit-framework.md)) is the mechanical layer that prevents template/source drift. This ADR adds two contracts on top of that hook:

#### D5a. Schema-derived enums regenerate atomically with their source

A schema edit that changes a closed enum referenced by `PLACEHOLDER_MAPPINGS` is, by the existing pre-commit-hook trigger, accompanied by a regenerated `.github/ISSUE_TEMPLATE/` in the same commit. The hook's `files:` regex (`^(scripts/TEMPLATES/.*\.yml|risk-map/schemas/.*\.schema\.json|risk-map/yaml/frameworks\.yaml)$`) already covers this. This ADR records the dependency so that a future contributor reading the hook configuration understands that the `risk-map/schemas/` trigger is load-bearing for the closure rule in D3.

#### D5b. CI in-sync check

CI runs `python scripts/generate_issue_templates.py` and `git diff --exit-code .github/ISSUE_TEMPLATE/` as a pull-request gate. Local hook-skips (`git commit --no-verify`) and contributors editing `.github/ISSUE_TEMPLATE/` directly are caught at CI rather than at merge. The check is cheap (generator is sub-second) and forecloses the failure mode where a generated template is hand-edited and the next regeneration silently overwrites it.

The CI step is registered against the same workflow that runs other pre-commit-equivalent checks per [ADR-024](024-github-actions-pinning-posture.md) pinning posture. Workflow pin form and lint compliance are inherited; this ADR does not re-author them.

### D6. Test discipline

The existing test suites — `scripts/hooks/tests/test_validate_issue_templates.py` and `scripts/hooks/tests/test_regenerate_issue_templates.py` — extend to cover the eight-source / eight-output post-D2 state. Specifically:

- Per-source structural fixtures: each of the eight source files has a regeneration-determinism test (run the generator twice; outputs are byte-identical) and a schema-conformance test (the generated YAML parses as a valid GitHub issue form).
- Placeholder-coverage test: `PLACEHOLDER_MAPPINGS` is asserted to cover every closed-enum schema field that any source template references. The list of referenced placeholders is extracted by parsing source files for `{{NAME}}` tokens; the list of closed enums is extracted from the schemas. A new closed-enum field added to a schema without a corresponding placeholder entry fails this test.
- Orphan-prevention test: the set of names returned by `IssueTemplateGenerator.get_available_templates()` equals the set of content-entity output files under `.github/ISSUE_TEMPLATE/` (excluding `infrastructure.yml` and `config.yml`, which are explicitly listed as out-of-scope by D1). A new generated content-entity output file without a matching source fails this test.

The orphan-prevention test is the mechanical guard against re-introducing the source-coverage gap that motivated this ADR.

### D7. Lifecycle and review

Template edits divide into two classes with different review profiles, and they target different branches per [ADR-002](002-branching-strategy.md):

| Class | Base branch | Rationale |
|---|---|---|
| Source-file structural changes — adding/removing fields, changing field types, adding/removing source files, updating `PLACEHOLDER_MAPPINGS` | `main` | Tooling change. Engineering review; small PRs land as they pass. |
| Source-file content changes — helper text, example placeholders, style-guide cross-references, ADR-citation manifest comments | `develop` | Contributor-facing teaching content. Community-review window matches the cadence of the contributing-guide pages the templates link to. |

The split mirrors [ADR-002](002-branching-strategy.md)'s framing of mixed PRs as "discouraged; split it." A PR that adds the `{{COMPONENT_SUBCATEGORIES}}` placeholder is a `main`-base tooling PR; a PR that retires HTML-link helper text from prose-field guidance is a `develop`-base content-adjacent PR. Implementation phases following this ADR sequence the two classes accordingly.

## Alternatives Considered

- **Document the orphan list and accept manual maintenance.** Rejected. The audit established that the orphans are guaranteed to drift on every schema enum change because the generator's coverage is silently partial. Documenting the asymmetry would freeze the drift class into the contract rather than closing it. The cost of backfilling three source files is bounded (≈½ day) and the maintenance cost of three orphans across an unbounded future is not. The repository's existing rule "edit `scripts/TEMPLATES/*.template.yml` sources first; the generated outputs are wiped on regeneration" is also more honest after backfill — today the rule is true for five templates and false for four, with no way for a contributor to tell which is which without reading generator code.

- **Restrict `scripts/TEMPLATES/` to non-orphan content templates only; document hand-edits as a permanent class.** Rejected. The proposed boundary (which templates are source-tracked vs. hand-edited) would be drawn case-by-case rather than by the structural rule "has a content schema." Without the structural rule, a future schema addition has no automatic answer to "does this template get a source file?", and the same audit-and-backfill cycle repeats. D2 draws the boundary by schema presence, which is mechanical and stable.

- **Move all template editing into a single hand-edit surface (delete the generator entirely).** Rejected. The generator solves a real coordination problem: closed-enum schema changes must propagate into contributor-facing intake forms, and a hand-edit surface guarantees drift between schemas and templates. Five templates have been generated for long enough that their schema-derived enums are reliably current; deleting the generator would re-open that drift surface and discard the existing test suite around it. The audit's verdict was that the generator under-covers, not that it should be removed.

- **Add closed-enum coverage only at the moments a contributor proposes a new template field.** Rejected as a closure rule. Reactive coverage means "discover the gap at the next audit." D3 is proactive: the placeholder registry is the source of truth for closed-enum solicitation; a source file that references an unregistered placeholder fails generator render. The mechanical layer (D6 placeholder-coverage test) catches the gap at PR time, not audit time.

- **Bind template helper text to ADR sections by automated cross-link.** Rejected for this ADR. An automated "this template teaches ADR-XXX D-section" graph (e.g., a `<!-- adr: ADR-017 D1 -->` annotation in helper text that a linter resolves) is an attractive idea but it would require building and maintaining new tooling that competes with the prose-lint surface (ADR-017 D4) for review attention. D4a's informational ADR-citation manifest at the top of each source file is the low-cost version: reviewers of an ADR amendment can mechanically grep for the ADR number across `scripts/TEMPLATES/` to find the templates that need to be re-read. If the manifest proves insufficient in practice, a follow-up ADR can promote it to a parsed annotation.

- **Treat the audit as a content-debt issue and fold into a Phase C wave.** Rejected as wrong-shape for the orphan-backfill and closure-rule portions. The content edits (retire HTML helper text; add `externalReferences[]` solicitation; retire `relevant-questions`; fix STRIDE/NIST/OWASP placeholders) are content-adjacent and fit a Phase-C-style sweep, but the orphan backfill and the `{{COMPONENT_SUBCATEGORIES}}` placeholder addition are generator-architecture changes that need an ADR-level commitment to "every content-entity-type has a source file" and "every closed-enum solicitation goes through `PLACEHOLDER_MAPPINGS`." Splitting this ADR into "architecture decision lands first; content sweep follows" keeps both pieces decision-bound rather than symbol-bound, per the repository's scope-by-decision rule.

## Consequences

**Positive**

- The source-coverage contract (D2) closes the silent drift surface that schema enum changes have today. After backfill, every closed-enum schema change propagates to all eight content-entity templates uniformly. The "edit sources first" rule becomes universally true rather than true-for-five-of-nine.
- The closed-enum closure rule (D3) prevents the `component-subcategory` class of bug from recurring. New schema enums are surfaced as placeholder registrations before any template solicits them; the order is enforced by the generator.
- The ADR-content alignment contract (D4) makes the chain `template → linked style guide → ADR` mechanically inspectable. A reviewer of an ADR amendment can grep `scripts/TEMPLATES/` for the ADR number and find every template that teaches the changed convention. A reviewer of a template change can read the manifest at the top of the file and check that the helper text matches.
- The test discipline (D6) catches three structural regressions at PR time: orphan re-introduction, placeholder gaps, and regeneration non-determinism. None of these are caught today.
- The CI in-sync check (D5b) forecloses the failure mode where a contributor edits a generated template directly (hand-edit silently overwritten by the next regeneration). Local hook-skips (`--no-verify`) are surfaced at PR time rather than at merge.

**Negative**

- Three new source files (`new_persona.template.yml`, `update_persona.template.yml`, `update_component.template.yml`) plus the eight existing sources are now maintained as a single class. Each source addition costs a small recurring maintenance increment (≈30 min per source per recurring ADR amendment that touches an entity). The increment is bounded and explicit, where the orphan-drift increment is unbounded and silent.
- The ADR-citation manifest at the top of each source (D4a) is human-curated documentation that can itself drift from the actual conventions the helper text teaches. The contract is informational, not mechanical, so a stale manifest is not caught by tests. Reviewers of helper-text edits must update the manifest in the same change; this is review discipline, not test discipline.
- The two-class lifecycle (D7) requires contributors to decide whether a template edit is structural or content-adjacent before opening a PR. The line is not always crisp: an edit that adds `externalReferences[]` solicitation is structural (a new field) and content-adjacent (new helper text describing the citation shape). Convention: a PR that touches both classes lands the structural half on `main` first, then the content-adjacent half rebases onto a `develop` that has absorbed the structural change. This mirrors the existing convention for split tooling/content PRs.
- The CI in-sync check (D5b) adds a fail-on-drift gate that contributors using `git commit --no-verify` will hit at PR time. The friction is intentional, but it is a new failure mode for contributors used to skipping local hooks.
- The closure rule in D3 commits the repository to maintaining `PLACEHOLDER_MAPPINGS` as the single registry for closed-enum solicitation. A schema change that closes an enum previously rendered as a textarea must now also register a placeholder; the change becomes a two-file coordination instead of a one-file edit. This is the correct trade for preventing the `component-subcategory` class of bug, but the coordination cost is real.

**Follow-up**

- Implementation phase 1 (architect-routed → swe + testing): backfill three source files; add `{{COMPONENT_SUBCATEGORIES}}` placeholder to `PLACEHOLDER_MAPPINGS`; extend `test_regenerate_issue_templates.py` and `test_validate_issue_templates.py` with the new fixtures plus the placeholder-coverage and orphan-prevention tests from D6. Target `main` per D7. This phase is the contract enforcement; downstream content phases consume it.
- Implementation phase 2 (content-reviewer): retire HTML-link helper text in all eight templates per ADR-017; add `externalReferences[]` textarea + sentinel-grammar guidance per ADR-016 D3; retire `relevant-questions` field/option from risk templates per ADR-019 D6; update STRIDE/NIST AI RMF/OWASP placeholders to canonical form per ADR-022 D5b; add `mappings` and `responsibilities` solicitations to component and persona templates per ADRs 018/021. Target `develop` per D7. Gated on phase 1 landing.
- Implementation phase 3 (content-reviewer): update the linked contributing-guide pages (`submission-readiness-guide.md`, `common-review-findings.md`, the four `*-titles-style-guide.md` pages, `framework-mappings-style-guide.md`, `identification-questions-style-guide.md`, `issue-templates-guide.md`, `template-sync-procedures.md`) to teach the post-Phase-B conventions templates now cite. Target `develop`. Can run parallel to phase 2.
- Register the CI in-sync check (D5b) as a new workflow or a new step in an existing workflow. The choice of surface is a small follow-up; this ADR does not pin it.
- If a future ADR adds a fifth content-entity type to the schemas, that ADR amends D2 (the four-entity-type list) and the implementation lands a fifth pair of source files plus the `entity_mappings` table entry. The amendment is mechanical; the contract shape does not change.

## Amendment 2026-05-21: Component category/subcategory valid-tuple selector

**Status:** Accepted (2026-05-21). Does not alter the Accepted status of D1–D7 above.
**Authors:** Architect agent, with maintainer review.

### Context

[D3](#d3-schema-derived-enum-contract--closed-enum-closure-rule) closed the components-subcategory `id` enum with a `{{COMPONENT_SUBCATEGORIES}}` placeholder, turning the template's free-form subcategory `input` into a dropdown of all seven subcategory IDs. That dropdown is **context-free**: it renders independently of the category dropdown, so a contributor can submit a category/subcategory combination that does not exist in the taxonomy (e.g. category `componentsApplication` + subcategory `componentsData`).

The combination matters because the valid pairings are a clean function — each subcategory belongs to exactly one category — but the name prefixes mislead:

| Category | Valid subcategories |
|---|---|
| `componentsInfrastructure` | `componentsData`, `componentsModelDeployment` |
| `componentsModel` | `componentsModelTraining`, `componentsModelCore`, `componentsOrchestration` |
| `componentsApplication` | `componentsAgent`, `componentsApplicationCore` |

`componentsModelDeployment` is an Infrastructure subcategory (not Model) and `componentsOrchestration` is a Model subcategory (not Application); the prefixes invite wrong guesses.

The deeper finding is that the form is not the only unguarded path. `components.schema.json` declares `component.category` (`#/definitions/category/properties/id`) and `component.subcategory` (`#/definitions/subcategory/properties/id`) as two independent scalar `$ref`s to flat enums, with no `allOf`/`if`/`then`/`dependentSchemas` coupling them. An invalid pair passes `check-jsonschema` and lands as bad content regardless of how it was authored — through the form, a hand-edit, or a generated diff. The form fix alone leaves the schema permissive.

This amendment **supersedes the interim `{{COMPONENT_SUBCATEGORIES}}` realization of D3's Immediate-scope paragraph.** D3's principles — single `PLACEHOLDER_MAPPINGS` registry, render-time closure of closed-enum solicitations — are unchanged and govern below. The amendment changes only the *placeholder model* (flat enum → derived join) and adds the schema-layer enforcement D3 did not reach. The interim placeholder was a coherent step: the closure rule landed first, the pairing constraint follows once the taxonomy join is modeled. History stays coherent — this is `amends`, not `supersedes ADR-026`.

Base branch is `main` per [D7](#d7-lifecycle-and-review) (structural: a field-shape change plus a generator-mechanism change plus a schema change). Downstream of acceptance is the standard test-first implementation chain.

### D8. Tuple-selector placeholder — amends D3's placeholder model

**Decision.** A `PLACEHOLDER_MAPPINGS` entry MAY resolve as a **derived join over the taxonomy** rather than a single flat enum. The subcategory solicitation becomes one such join:

- Retire `{{COMPONENT_SUBCATEGORIES}}` (the flat seven-value dropdown D3 added).
- Add `{{COMPONENT_CATEGORY_SUBCATEGORY}}`, a dropdown whose options are the valid `category-id: subcategory-id` tuples — seven options, one per existing pairing. GitHub issue forms have no cascading/dependent dropdowns, so a flattened valid-tuple list is the only form-fill-time enforcement available; this is the canonical workaround, not a compromise.

```
- componentsInfrastructure: componentsData
- componentsInfrastructure: componentsModelDeployment
- componentsModel: componentsModelTraining
- componentsModel: componentsModelCore
- componentsModel: componentsOrchestration
- componentsApplication: componentsAgent
- componentsApplication: componentsApplicationCore
```

**This is a resolver-kind, not a parallel system.** `PLACEHOLDER_MAPPINGS` remains the single registry D3 requires. The join is a new resolver branch beside the existing `FRAMEWORK_MAPPINGS` special-case in [`template_renderer.py`](../../scripts/hooks/issue_template_generator/template_renderer.py) (`expand_placeholders` already branches on placeholder name before the generic flat-enum path, and `yaml_source`-bearing entries like `PERSONAS` already read YAML data). The closure rule and render-time-failure-on-unregistered-placeholder semantics of D3 carry over unchanged.

**Option format is ID:ID, delimiter `": "`.** Each option is `<category-id>: <subcategory-id>`. The dropdowns already solicit raw entity IDs; friendly titles would introduce a title→ID inconsistency and a reverse-lookup map. Splitting an option on `": "` yields the `category` and `subcategory` field values directly.

**Parse-back is a documented convention, not built tooling.** Issue → YAML transcription is manual or agent-mediated today; no automated pipeline exists. The `": "` split is recorded as the convention a transcriber applies; no parser is built (YAGNI). If an automated issue → YAML pipeline is ever introduced, it inherits this convention as its contract.

### D9. Generator trigger and read-set widening

**Decision.** Add `risk-map/yaml/components.yaml` to the `regenerate-issue-templates` hook `files:` trigger in [`.pre-commit-config.yaml`](../../.pre-commit-config.yaml).

The authoritative tuple source is the category→subcategory **nesting** declared in `components.yaml` (`categories[].subcategory[]`), not the schema. The schema's `definitions.category` carries a `subcategory` array property — the *shape* that a category may nest subcategories — but it does not enumerate which subcategories belong to which category; that assignment is data in `components.yaml`. Tuples are therefore derived from the taxonomy nesting, **not** from component instances (`components[]` — a zero-instance subcategory would silently vanish) and **not** from ID name prefixes (they mislead, per Context).

The current trigger (`^(scripts/TEMPLATES/.*\.yml|risk-map/schemas/.*\.schema\.json|risk-map/yaml/frameworks\.yaml)$`) does not match `components.yaml`, so a taxonomy edit would not regenerate the affected template. Widening the trigger keeps the generated output atomic with its taxonomy source, matching [D5a](#d5a-schema-derived-enums-regenerate-atomically-with-their-source).

This fits the [ADR-005](005-pre-commit-framework.md) Addendum 2026-05-08 generator carve-out, which holds `regenerate-issue-templates` outside the trigger-vs-read-set invariant because generators are side-effectful with read sets spanning multiple source YAMLs. The generator already reads `personas.yaml` (deprecated-ID filtering) without a dedicated trigger entry under that carve-out; reading the `components.yaml` taxonomy is the same precedented capability. No new ADR-005 treatment is required — this amendment records the decision and cites the addendum.

### D10. Schema conditional pairing constraint — the content-integrity fix

**Decision.** Add a conditional pairing constraint to `definitions.component` in [`components.schema.json`](../../risk-map/schemas/components.schema.json) so an invalid `(category, subcategory)` pair is **rejected by `check-jsonschema` on every edit path**, not only at the form. The form selector (D8) is a usability guard; this constraint is the enforcement.

The shape is an `allOf` of one `if`/`then` per category, each `then` restricting `subcategory` to that category's valid subset. Representative clause:

```json
{
  "if":   { "properties": { "category": { "const": "componentsInfrastructure" } } },
  "then": { "properties": { "subcategory": {
              "enum": ["componentsData", "componentsModelDeployment"] } } }
}
```

`subcategory` is not a required component property (component `required` is `["id", "title", "category", "edges"]`); `then.properties.subcategory` only constrains the value when present, so components that omit a subcategory stay valid. The flat `definitions.subcategory.properties.id` enum is retained as the per-value type — the `allOf` narrows it conditionally rather than replacing it.

This is the load-bearing change. With it, the bad-pair class is closed regardless of authoring path; without it, the selector would be the only barrier and a hand-edit or generated diff could still introduce an invalid pair under a green CI.
