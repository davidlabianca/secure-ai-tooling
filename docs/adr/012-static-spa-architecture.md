# ADR-012: Static client-side SPA — no backend, vanilla ESM, `node --test`, progressive-enhancement a11y

**Status:** Accepted
**Date:** 2026-04-21
**Authors:** Architect agent, with maintainer review

---

## Context

The persona Pages MVP (PR #223) had to ship a persona-matching explorer onto the deploy surface established by [ADR-009](009-persona-pages-workflow-topology.md): GitHub Pages, static hosting, OIDC deploy, no server side. The PR #223 review captured the resulting architectural posture in §1 — the MVP is *"zero-backend, zero-storage of answers, zero framework churn for YAML content"* — and treated that posture as a positive invariant to preserve, not an implementation default to be filled in later.

Three consumer-side shapes fell out of that posture together in the MVP's initial commits (`48565ca` / `e0dd0bb` / `0ccd701`) and were hardened in-PR through the review's implementation plan:

1. **Zero backend, zero telemetry.** `site/index.html` is plain HTML, `site/assets/styles.css` is plain CSS, and two ES modules (`app.mjs`, `persona-logic.mjs`) drive the page. Persona answers live only in the in-memory `state` object inside `app.mjs`; there is no `fetch` back to an API, no `localStorage` / `sessionStorage`, no analytics, no cookies. The `<noscript>` fallback in `index.html:38` states the contract to visitors directly: *"The CoSAI Risk Map Explorer keeps answers in your browser and does not use a backend."*
2. **Vanilla ES modules, zero bundler, `node --test` for pure-logic tests.** No React/Vue/Svelte, no Vite/Webpack/esbuild, no `package.json` devDependency for a test framework. The renderer uses native `<script type="module">` loading; tests use Node's built-in `node:test` + `node:assert/strict`. Review §6.2 recorded this as a deliberate first — `persona-logic.test.mjs` is *"the first Node test file in the repo (confirmed by grep for `node --test`, `jest`, `mocha`, `vitest`)"* — and §7.1 recorded the reason to keep it that way: *"`package.json` currently has only `prettier` and `@mermaid-js/mermaid-cli`. Adding vitest/jest would be the first test framework in the repo; every contributor pays the install cost forever."*
3. **Progressive-enhancement a11y layer.** Commit `9de0390` ("feat: site renderer UX and a11y layer 1") landed the aria-live status region, `announceStatus()` helper, `.visually-hidden` utility, `:focus-visible` outlines, and `prefers-reduced-motion` handling — the findings tagged `NIT-12`, `REC-19`, and `REC-20`. The scope was consciously bounded: `NIT-12`'s verdict was *"APPLY — layer 1 only"* and focus-management after DOM patches was explicitly moved into the `NIT-07` follow-up (tracked as [issue #224](https://github.com/cosai-oasis/secure-ai-tooling/issues/224)), not deferred by accident.

These three shapes share one forcing constraint. GitHub Pages serves static bytes; the privacy posture the review endorsed ("zero-storage of answers") forbids a server-side component even if one were cheap; and the repo's existing `package.json` surface (two dependencies, both for tooling) makes a new framework a *per-contributor* install cost forever. Taken together they are a single architectural commitment — *how the renderer is shaped* — and this ADR documents them as one. The downstream contract the renderer consumes ([ADR-011](011-persona-site-data-schema-contract.md)), the module boundary it lives inside ([ADR-010](010-site-repo-root-module-boundary.md)), and the deploy surface that hosts it ([ADR-009](009-persona-pages-workflow-topology.md)) are separate decisions; this ADR covers only the renderer architecture.

One later commit — `b5514cb` ("feat(site): port Shrey's loading-card polish and status-card helper") — extracted a `renderStatusCard` helper so the loading and error paths share a single card component, and unified the three-bullet status list between `index.html` and the JS loading state. It is UI polish that fits the pattern above (extraction as the renderer found a second caller) rather than a new architectural move, and is cited only as evidence that the renderer's "extract when a second caller appears" posture is alive.

This ADR documents the renderer architecture retroactively, under the practice established by [ADR-001](001-adopt-adrs.md).

## Decision

The persona Pages surface is a **static, client-side-only single-page application** served from GitHub Pages. Three sub-commitments share the single motivating constraint above and are treated as one decision:

### 1. Zero backend, zero telemetry

- **All rendering is browser-side.** `site/index.html` loads `site/assets/app.mjs` as a native ES module (`<script type="module">`). `app.mjs` fetches the generated JSON artifact — `./generated/persona-site-data.json` — via `fetch` and renders the page with `innerHTML` assignments into a single `<main data-app>` element.
- **Persona answers never leave the browser.** State lives in the `state` object at `site/assets/app.mjs:7-17` (`answers`, `manualSelectedIds`, `personaOverrides`, `step`, `activeTab`). The app never writes to `localStorage`, `sessionStorage`, `IndexedDB`, cookies, or any remote endpoint. The only `fetch` call is the initial read of the schema-validated data artifact produced by `scripts/build_persona_site_data.py`.
- **No analytics layer.** No third-party script tag, no measurement endpoint, no beacon. The privacy-badge copy at `app.mjs:677` ("Answers stay in this browser session only") and the `<noscript>` copy at `index.html:38` are the user-facing statements of this invariant; the code upholds them.
- **No server-side rendering.** The initial HTML shipped to the browser is a loading card (`index.html:17-33`); content is materialized after the client fetches and validates the JSON artifact. Crawlers that execute JavaScript will see the full page; crawlers that do not will see the loading card and the `<noscript>` statement.

### 2. Vanilla ES modules, zero bundler, `node --test`

- **Module split: pure logic vs. DOM rendering.** `site/assets/persona-logic.mjs` exports pure input→output functions (`dedupeInOrder`, `getSelectedPersonas`, `buildResultsModel`). `site/assets/app.mjs` imports them and holds everything DOM-touching — event listeners, state mutation, `innerHTML` writes. The split is present from the MVP's initial commit (`48565ca`) and is the property that makes `persona-logic.mjs` testable without a DOM.
- **No framework, no bundler, no transpiler.** The browser loads `.mjs` files directly. `.prettierrc.yml` carries the per-file override that makes `site/**/*.{mjs,js}` use `singleQuote: false, semi: true, trailingComma: 'all'` (standard JS convention) while keeping the repo-wide YAML defaults intact — one formatter, two style contexts.
- **Tests run under `node --test`.** `site/tests/*.test.mjs` uses `node:test` + `node:assert/strict`, invoked as `node --test site/tests/*.test.mjs`. No Jest, Vitest, or Mocha. The authoritative doc is `site/tests/README.md` (landed alongside `9de0390`'s Phase 4 work), which codifies the zero-install property and names the conditions under which a framework could be introduced (an "Escalation" procedure requiring a concrete case not expressible under `node --test`, a pinned dependency, a workflow update, and a README update in the same PR).
- **`package.json` remains tooling-only.** Today's `package.json` dependencies (`prettier`, `@mermaid-js/mermaid-cli`) are repo-wide formatters; adding a test framework would be the first runtime-test dependency and is governed by the Escalation process above.

### 3. Progressive-enhancement a11y — layer 1

- **Persistent live region.** `<div id="app-status" class="visually-hidden" role="status" aria-live="polite" aria-atomic="true">` lives in `site/index.html:16` outside `<main data-app>` so it survives `innerHTML` replacement during re-render.
- **`announceStatus(message)` helper.** Defined at `app.mjs:35-40`; called on step navigation (`:719`), answer change (`:749`), manual persona toggle (`:761` / `:766`), and session reset (`:737`).
- **Focus/reduced-motion styling.** `:focus-visible` outlines on all interactive elements and an `@media (prefers-reduced-motion: reduce)` block in `site/assets/styles.css`; `scrollTo({ behavior: ... })` at `app.mjs:720-723` guards on `matchMedia("(prefers-reduced-motion: reduce)").matches` to pick `"auto"` vs. `"smooth"` per run.
- **`.visually-hidden` utility.** Added to `styles.css` for the live region and reusable by any future off-screen label.
- **Consciously bounded to layer 1.** Focus-management after DOM patches (preserving focus across `innerHTML` writes) is *not* part of this ADR's scope — the review verdict on `NIT-12` is explicitly *"APPLY — layer 1 only"* and the follow-up is filed as part of the `NIT-07` draft issue. This is a documented scope line, not an oversight.

## Alternatives Considered

- **React / Vue / Svelte + bundler.** Rejected. The renderer has no routing, no shared state library, no hydration story to justify. A bundler introduces a Node toolchain that every contributor installs, a build step between `git pull` and "open the page in a browser," and a framework lifecycle (upgrades, breaking changes, ecosystem churn) the MVP does not need. The review's §1 summary already captured the positive posture as "zero framework churn for YAML content"; adopting a framework here would invert that posture for the consumer.
- **Server-rendered or API-backed renderer.** Rejected. Contradicts the privacy posture ("zero-storage of answers" per the PR #223 review §1): a server-side renderer would need to receive persona answers to return personalized HTML, introducing a trust surface the MVP deliberately does not have. Also requires hosting infrastructure beyond GitHub Pages, which the deploy surface in ADR-009 does not provide.
- **Analytics / telemetry backend.** Rejected. "Answers stay in this browser" is a review-endorsed posture, not a default that ought to be filled in later. Even anonymous session telemetry would require a beacon endpoint and a third-party script, both of which cross the privacy line. If future product needs change this, the change is an ADR-level revisit, not a silent script-tag addition.
- **Jest / Vitest / Mocha as the test runner.** Rejected. Review §7.1 recorded the concrete reason: *"Adding vitest/jest would be the first test framework in the repo; every contributor pays the install cost forever."* `node --test` is stable in the pinned Node 22 (`.mise.toml`), covers the pure-logic surface today, and `site/tests/README.md` documents an explicit Escalation path for the moment the surface changes shape (DOM integration, coverage thresholds, browser harness). The current zero-dep property is easier to preserve than to reclaim.
- **Full WCAG audit / a11y layer 2+ in-PR.** Rejected as in-PR scope. `NIT-12`'s verdict is explicitly *"APPLY — layer 1 only"*; focus-management after DOM patches, keyboard-shortcut discoverability, high-contrast mode, and screen-reader-specific testing are tracked in [issue #224](https://github.com/cosai-oasis/secure-ai-tooling/issues/224) (the `NIT-07` follow-up). Layer 1 (structural: live region, focus indicators, reduced-motion) is a deliberate MVP posture that any later audit layers on top of, not a compromise waiting for a cleanup PR.
- **Eager HTML-string extraction from `app.mjs`.** Rejected as in-PR scope. `app.mjs` has zero exports today, so every `renderXxx` function is module-private and unreachable from `node --test` without either a refactor or a DOM emulator. `NIT-07`'s verdict is *"DEFER — draft issue authored"*; the refactor path (extract pure HTML builders into `site/assets/app-render.mjs`) is named as the recommended follow-up. Doing the refactor in the MVP PR would have conflated a measurable architectural change with the ship-to-Pages blocker; deferring it preserves the "consumer of framework data, ready to extract when a second caller appears" posture visible in `b5514cb`'s `renderStatusCard` extraction.

## Consequences

**Positive**

- **The privacy posture is architecturally enforced, not just documented.** There is no server to leak answers from, no storage layer to forget to clear, and no analytics script to silently accumulate telemetry. A contributor cannot accidentally add a beacon without a visible code change (new `fetch`, new script tag) that review would catch.
- **Zero-install local preview.** `python3 scripts/build_persona_site_data.py` + any static file server is the entire "run it locally" surface. No `npm install` for the renderer, no bundler warm-up, no watch process — the `.mjs` the browser executes is the `.mjs` on disk.
- **Zero-dep test surface.** `node --test site/tests/*.test.mjs` runs with only the pinned Node toolchain (`.mise.toml`). CI pays no install cost for a test framework; contributors pay no install cost to reproduce CI. The test-convention doc (`site/tests/README.md`) is the contract future contributors edit against.
- **Pure-logic isolation is testable.** `persona-logic.mjs` is deterministic input→output with hand-built fixtures; persona matching, de-duplication, and category grouping are all exercised without a DOM. The module split that makes this possible is the same split the `NIT-07` follow-up builds on (extracting `app-render.mjs` from `app.mjs` is additive, not a rewrite).
- **Progressive-enhancement baseline is in place.** The aria-live region survives `innerHTML` re-renders because it sits outside `<main data-app>`; `:focus-visible` and `prefers-reduced-motion` are honored from layer 1; future audit layers add to the same scaffolding rather than replacing it.

**Negative**

- **`app.mjs` has zero exports → completely untested.** Review finding `NIT-07` is the canonical statement of this cost: the renderer's HTML-string builders, event handlers, and state mutations are all module-private, so `escapeHtml`, `renderRichParagraphs`, `renderQuestionGroups`, `renderResults`, and every other `renderXxx` function is unreachable from `node --test` today. A regression in any of them — including an XSS hole in `escapeHtml` — is only caught by manual inspection. The recommended path (extract into `site/assets/app-render.mjs`, test directly) is a named follow-up, not a commitment of this ADR.
- **`innerHTML` re-render destroys focus.** Every state change wipes `<main data-app>` and rebuilds it; the focused node is replaced, and focus drops to `<body>`. The aria-live region mitigates the screen-reader impact, but keyboard users lose their place on every click. Focus-management after DOM patches is explicitly layer-2 work and is part of the `NIT-07` follow-up issue's acceptance criteria.
- **No SSR, no crawler fallback beyond the loading card.** Non-JavaScript clients see the loading card and the `<noscript>` statement; search engines that do not execute JavaScript will not index the persona content. This is an acceptable trade for a private-by-construction explorer, but it is a trade: if the site is later expected to be indexed by non-JS crawlers, the renderer architecture is what changes.
- **`innerHTML` string-templating relies on `escapeHtml` discipline at the trust boundary.** `escapeHtml` (defined at `app.mjs:21-33`) is the only defense between untrusted prose (YAML-sourced strings flowing through the schema) and the DOM. ADR-011's schema constrains the shape (`oneOf: [string, array<string>]`) but does not constrain string contents — a YAML author can legitimately write HTML tags in `longDescription` fields by design (see review finding REC-10). Every new renderer function must remember to route user-facing strings through `escapeHtml`, and every reviewer must check. The alternative — a templating library with contextual auto-escaping — would re-introduce the framework dependency this ADR declines.
- **A11y is layer 1 only.** The MVP ships structural accessibility (live region, focus indicators, reduced-motion) but not behavioral accessibility (focus preservation, keyboard shortcuts, high-contrast, screen-reader flow testing). Contributors reading the code may misread the presence of `aria-live` as a finished a11y story; `site/tests/README.md` and the `NIT-07` draft issue are the counter-weights, but the gap is real.
- **Tooling supply chain is narrow but pinned.** The renderer depends on browser-native ESM, browser-native `fetch`, and a Node runtime that supports `node:test` (pinned to Node 22 via `.mise.toml`; `node:test` is stable since Node 20, so the narrowness is the pin rather than the feature). That is a small surface, but a drop in Node-pin currency or a browser-support regression in `<script type="module">` handling would affect tests or runtime respectively. The narrowness makes this easy to audit; it does not make it zero.

**Follow-up**

- **`NIT-07` — `app.mjs` testability.** Tracked as [issue #224](https://github.com/cosai-oasis/secure-ai-tooling/issues/224). Recommended path: extract pure HTML builders into `site/assets/app-render.mjs`, add `site/tests/app-render.test.mjs`, keep the zero-dep property. Focus-management after DOM patches is part of that issue's acceptance criteria, not a separate ticket.
- **XSS defense options beyond `escapeHtml` discipline.** Four additional layers were scoped during this ADR's review and are available but not committed here; each is tracked as a separate follow-up: (a) allowlist validation of YAML prose HTML at pre-commit (reject-and-block on `<script>`, `javascript:`, `on*=`, etc., using `nh3` under the ADR-013 hook pattern); (b) sink elimination for scalar fields rolled into the NIT-07 refactor (issue #224) so `innerHTML` is reserved to `renderRichParagraphs` alone; (c) a meta-CSP backstop in `site/index.html` (`default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'; frame-src 'none'`); (d) direct unit tests on `escapeHtml` closing the untested-helper gap above. Trusted Types enforcement via `require-trusted-types-for` is explicitly **out of scope** on the GitHub Pages deploy surface — the directive is HTTP-header-only in current browsers and `<meta http-equiv>` silently drops it; a move to a deploy surface that serves custom response headers ([ADR-009](009-persona-pages-workflow-topology.md) revisit) is the trigger.
- **A11y layer 2+.** Focus preservation across re-renders, keyboard shortcut discoverability, high-contrast mode, and screen-reader-specific flow testing are follow-ups. Each is a separate issue; the `NIT-07` refactor is the prerequisite for focus-management because the renderer has to become testable before focus behavior can be regression-tested.
- **Renderer consumes the ADR-011 contract.** The schema shape in [ADR-011](011-persona-site-data-schema-contract.md) — specifically `definitions/prose`'s `oneOf: [string, array<string>]` — is the exact shape `renderRichParagraphs` (`app.mjs:68-78`) is written to handle. A future schema change is a renderer change; the coupling is one-directional (renderer tracks schema) and is what ADR-011's Follow-up already flagged. Commit-time enforcement of that contract — the pre-commit hook that runs the builder before push, so a renderer-incompatible output is caught locally — is the subject of [ADR-013](013-site-precommit-hooks.md).
- **`renderStatusCard` as extract-when-needed evidence.** Commit `b5514cb` is the first instance of the renderer extracting a helper when a second caller appeared (loading + error paths). Future polish should follow the same posture — extract locally when a second caller shows up; do not pre-factor for hypothetical callers. The `NIT-07` refactor is a larger instance of the same pattern.
- **If analytics or preview deploys are ever proposed,** they are ADR-level revisits, not silent additions. The privacy posture and the framework-free renderer are both load-bearing here; a change to either is a change to this ADR.
- **Escalation path for a test framework is documented.** `site/tests/README.md`'s Escalation section names the procedure: concrete case not expressible under `node --test`, rejected alternatives, pinned dependency, workflow update, README update. Follow it rather than extending this ADR if that moment comes.
