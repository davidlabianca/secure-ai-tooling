# ADR-027: Per-mapping framework version pinning

**Status:** Accepted
**Date:** 2026-05-22
**Authors:** Architect agent, with maintainer review

---

## Context

The CoSAI Risk Map cross-walks its content against external frameworks declared in `risk-map/yaml/frameworks.yaml`: `mitre-atlas`, `nist-ai-rmf`, `stride`, `owasp-top10-llm`, `iso-22989`, `eu-ai-act`. Content entities (`risks`, `controls`, `components`, `personas`) reference them through a `mappings` object keyed by framework `id` whose values are taxonomy identifiers (`mappings.mitre-atlas: [AML.T0020]`). [ADR-022](022-supporting-schemas.md) D5b committed per-framework mapping-ID regexes in `risk-map/schemas/frameworks.schema.json` `definitions/framework-mapping-patterns` (commit `4ae3fe2`); [ADR-016](016-reference-strategy.md) D3 committed the referentially-adjacent `externalReferences[].id` canonical-form regexes.

Three coupled questions are resolved together here: the framework **versioning posture** (#303), **mapping-convention harmonization** across all frameworks (#318), and the **NIST AI RMF persona axis** policy (#319), filed as a trio under epic #342 with the content migration tracked separately as #343. They interlock — the convention shape (#318) depends on the versioning posture (#303), and the NIST axis (#319) must fit whatever convention #318 lands — so they take one ADR. The imminent registration of a seventh framework, ETSI EN 304 223 (#329, with persona-mapping proposals #273/#274 blocked on it), is folded in so it lands in the durable form rather than an interim one.

### The integrity problem

The central concern is not version *display* but mapping **integrity across framework re-versions**: which version of a framework does a mapping resolve against, and does that mapping still mean what it meant when it was authored? Two failure shapes drive it — **silent re-interpretation** (a stable-looking identifier means something different in a newer edition) and **silent dangling** (an identifier is retired or renamed and no validator notices). Three real-spec examples make the failure concrete:

- **OWASP LLM 2023 → 2025 (semantic shift).** `LLM02` is "Insecure Output Handling" in the 2023 edition but "Sensitive Information Disclosure" in the 2025 edition. The same identifier names a different threat across editions; a mapping that does not pin its edition is silently re-pointed when the registry version moves.
- **MITRE ATLAS technique retirement (dangling).** A value like `AML.T0043` carries no version. If a future ATLAS matrix retires or renumbers the technique, a registry-level "latest-wins" resolution leaves the mapping pointing at nothing, and `validate_framework_references.py` — which checks framework-registry *existence*, not the framework's technique catalog — does not catch it.
- **EU AI Act 2021-proposal → 2024-final (renumber).** The transparency obligations were **Article 52** in the 2021 Commission proposal and became **Article 50** in the 2024 final regulation. A value `Article 52` carries no version; if the registry entry is updated in place to the 2024 text, `Article 52` now resolves against a different provision — silently wrong.

The frameworks re-version on independent cadences (MITRE ATLAS quarterly; OWASP LLM Top 10 by-year; NIST AI RMF 1.0 with a 1.1/2.0 revision under way per airc.nist.gov; ISO 22989:2022 amendments expected; EU AI Act phased effective dates 2025-2027; STRIDE stable). `frameworks.yaml` carries a free-form `version` field (`'5.0.1'`, `'2025'`, `null`) but **nothing relates a mapping to the framework version it was authored against**, so the failure shapes above are un-detectable today. The gap is cheap to close before the first version-bump lands and expensive to retrofit after.

A precise statement of what version-pinning can and cannot guarantee is load-bearing for the rest of this ADR. Pinning the authored version into each mapping value makes drift **expressible** — every mapping records the edition it was authored against — and therefore makes the re-interpretation and dangling cases **detectable in principle**. The detection itself splits into two tiers (D5): a membership/version-known check that ships immediately, and a catalog-absence check that depends on per-version framework catalogs CoSAI does not yet hold. The honest near-term guarantee is therefore *version-known plus controlled-vocabulary membership*, not retire/renumber detection; the latter becomes possible once the pinned version exists to join against, but is a later enhancement, not an adoption-time guarantee.

### Convention divergence

The `framework-mapping-patterns` regexes carry divergent shapes (dotted ATLAS IDs, `FUNCTION-N.N` NIST subcategories, a six-value STRIDE enum, year-suffixed OWASP, `Article N(n)` EU references, and an ISO 22989 free-string carve-out). #153 (raymondsheh) proposed a CoSAI-defined structured form (`iso22989:2022:ai_customer:ai_user`) that embeds the framework version inside the value and generalizes to non-ID-bearing frameworks; #164 (reikawa13, prospective EU AI Act personas) treats its Article-reference form as interim pending this decision. OWASP's `^LLM\d{2}:\d{4}$` pattern *already* embeds the edition year in the value — it is the one framework whose committed pattern is integrity-safe by construction, and it is the prototype this ADR generalizes.

A free-string value for the non-ID-bearing frameworks is not an acceptable result for a standard: under `{"type": "string"}`, `AI Partner (data supplier)`, `AI Partner (datasupplier)`, and `AI Part (Data supplier)` all validate and nothing flags the divergence. A non-ID-bearing framework's roles/tasks are a *closed, enumerated set* published in the spec's own table, so the values are knowable today and can be validated against that closed set — the standardization #153's structured-form instinct was reaching for, scoped to the non-ID frameworks (the ID-bearing frameworks keep their spec-native regex, which already constrains them).

### NIST AI RMF persona axis (#319)

raymondsheh proposed mapping `personaModelServing` to five NIST AI RMF *Actor Tasks* (Appendix 3 of AI RMF 1.0: AI Deployment, Operation and Monitoring, TEVV, Procurement, Governance and Oversight). The current `nist-ai-rmf` D5b pattern `^(GOVERN|MAP|MEASURE|MANAGE)-\d+(\.\d+)*$` constrains values to *Subcategory IDs* — a different axis. AI RMF 1.0 assigns no canonical IDs to Actor Tasks; they are text-only. `frameworks.yaml` declares `nist-ai-rmf.applicableTo: [controls]` today; admitting personas requires both an `applicableTo` change and an axis policy.

### The migration baseline

The existing content already diverges from the D5b patterns: STRIDE values are lowercase-kebab (`tampering`, `information-disclosure`), OWASP values are un-versioned (`LLM04`, not `LLM04:2025`), and NIST values use the short `GV-`/`MS-`/`MP-`/`MG-` prefixes (e.g. `controls.yaml:541` `GV-6.2`, `risks.yaml:106` `tampering`, `risks.yaml:108` `LLM04`) rather than the long `GOVERN-`/`MEASURE-`/`MAP-`/`MANAGE-` forms the D5b regex requires. The consumer schemas (`risks`/`controls`/`components`/`personas`) deliberately fall through to a loose catch-all for these three frameworks, so the legacy values still validate; the D5b patterns are *defined but not yet enforced on content*. `framework-mappings-style-guide.md` teaches the legacy forms, so the docs and the D5b schema patterns are already inconsistent. Any convention decision lands on top of an *already-pending* migration, not a clean baseline.

The migrated unit is the individual mapping value, not the per-framework block. Across the three populated content files **46 entities carry mappings**, spread over **96 `mappings.<framework>:` blocks** (risks 65, controls 25, personas 6), and those blocks comprise **146 individual values** (risks 100, controls 40, personas 6). EU AI Act has **zero mapping content on disk today** — its only present-tense presence is the `frameworks.yaml` registry entry; EU mappings arrive *prospectively* through the unlanded #164. So EU AI Act is a prospective consumer of this convention, not part of the present-tense migration cost.

### Versioning prior-art and direction

Maintainer directional input (#303 comments) is a hybrid posture: keep a dedicated `version` field in `frameworks.yaml` as authored source of truth *and* make the referenced version visible in the identifier, not buried in a side field. The durable version-distinct identifier is *generated* by tooling from authored fields, never hand-picked — mirroring how DataCite mints DOIs and how this repo already generates templates and tables. The closest prior-art is the DataCite/DOI versioning model: a stable Concept DOI for the version family plus per-version DOIs, with the version number stored in a separate `<version>` metadata field and lineage expressed via `RelatedIdentifiers` (`IsNewVersionOf` / `IsPreviousVersionOf`).

## Decision

We adopt **per-mapping version pinning in the mapping value** as the integrity mechanism: a mapping records the framework version it was authored against *inside its value*, generalizing OWASP's existing year-in-value (`LLM02:2025`) across every versioned framework. The mapping **key stays the bare framework `id`**, so the `framework.id` enum, the `propertyNames` `$ref`, and the D5b key set are unchanged — there is no key collision. Pinned values are **generated, not authored**, by a `framework-mapping-maintainer.py` tool from a contributor's structured selection, paired with a purity validator and a drift-detection validator. Mappings remain **bare-string arrays**: the tool's `update`/`remove` verbs address a mapping **positionally** by its structured tuple, keyed internally by a **derived (non-stored) `mappingId`**, so the consumer schemas (`risks`/`controls`/`components`/`personas`) are unchanged (D4b). We **keep the registry `version` field and a generated `versionId`** as the source-of-truth version list the tool validates against and as the citation/lineage anchor. We **harmonize the mapping-value convention to spec-native canonical forms grown by one version dimension** (not a CoSAI-invented structured form). The NIST AI RMF Actor-Task axis is admitted as a **separate framework `id`**. For the **non-ID-bearing frameworks** (ISO 22989, the NIST Actor-Task axis, and ETSI EN 304 223) we **replace the free-string carve-out with a closed, version-pinned controlled vocabulary encoded as an inline per-version `enum`** in `framework-mapping-patterns`, sourced from each spec's own role/task table. We define a **version-change process** (D10) scaled to blast radius, so a framework re-issuance is a gated, owned operation rather than ad-hoc. We **register ETSI EN 304 223 (#329) in this ADR's form** (D11) as the first worked instance of that controlled vocabulary.

The cost is named plainly: the D5b per-framework *value* patterns grow a version dimension, the non-ID-bearing entries become closed inline enums, and the content mapping values — 96 `mappings.<framework>:` blocks comprising 146 individual values (risks 100, controls 40, personas 6) — migrate to pinned form. That migration is #343 execution, not this ADR. The mapping *keys*, the `framework.id` enum, the consumer schemas, and ADR-016 D3 are untouched (D7).

The immediate guarantee this delivers is *version-known plus controlled-vocabulary membership* (D5 Tier 1): every pinned value is checked against the framework versions the registry recognizes and, for the non-ID frameworks, against the closed enum for the pinned edition. Detecting that a *technique was retired or an article renumbered* requires per-version framework catalogs CoSAI does not yet hold; pinning makes that detection possible later (D5 Tier 2) but does not deliver it at adoption.

### D1. Version metadata location — dedicated registry field as source-of-truth, mappings pin per-value

Framework version metadata lives in `frameworks.yaml` on the framework entry as the **authoritative list of versions that exist** (the hybrid half, and the set the generator tool validates `--version` against). The existing `version` field holds the *current* version of each framework. The registry answers "what versions of this framework does CoSAI recognize, and which is current"; the mapping value (D3) answers "which version was *this* mapping authored against."

This is a deliberate split from registry-only inheritance (latest-wins), under which every mapping silently follows the registry forward — exactly what makes the ATLAS-retire and EU-renumber cases (Context) silently wrong. Per-value pinning (D3) breaks that coupling: the registry's `version` advancing no longer silently re-points existing mappings; it instead makes their pinned version *visible as superseded* to the drift validator (D5), which a maintainer reviews rather than the registry silently resolving.

### D2. Generated registry `versionId` and lineage — citation, display, supersession anchor

Each framework registry entry carries a **generated `versionId`** distinct from the stable concept `id` used as the mapping key. This is the DataCite Concept-DOI / version-DOI split: the concept `id` (`mitre-atlas`) is the stable family identifier; the `versionId` is the version-distinct instance identifier.

#### D2a. ID shape

- **Concept id (existing, unchanged):** the kebab framework key — `mitre-atlas`, `nist-ai-rmf`, `owasp-top10-llm`, `stride`, `iso-22989`, `eu-ai-act`. This is the closed enum the mapping keys, the D5b allowlist, and `validate_framework_references.py` already use. **It does not change** and remains the mapping key (D7).
- **versionId (new, generated):** `<id>@<version>` — the concept id, an `@` separator, and the normalized `version` string. Examples: `mitre-atlas@5.0.1`, `nist-ai-rmf@1.0`, `owasp-top10-llm@2025`, `eu-ai-act@2024`, `iso-22989@2022`. For unversioned frameworks (`version: null`, i.e. STRIDE) the `versionId` is the bare concept id (`stride`).

The `@` separator is chosen over a `-` suffix (`mitre-atlas-5.0.1`) deliberately: a dash suffix is ambiguous against kebab concept ids (a parser cannot tell `nist-ai-rmf-1.0` apart from a hypothetical concept id `nist-ai-rmf-1`), whereas `@` cannot occur in a concept id (closed kebab enum) or in any current `version` string, so the split is unambiguous and the concept id is recoverable by truncating at `@`. The shape echoes package-manager `name@version` (npm `pkg@1.2.3`) and is the per-value version token D3 reuses.

#### D2b. Generation rule and where it runs

`versionId` is **derived, never authored**: `versionId = id if version is null else f"{id}@{version}"`. A contributor edits only `id` and `version`; the generator materializes `versionId`. It runs as a **pre-commit generator hook** alongside the existing generated-artifact hooks (the table generator `yaml_to_markdown.py` and the diagram generators governed by [ADR-005](005-pre-commit-framework.md) / [ADR-013](013-site-precommit-hooks.md)), with a paired **purity validator** that fails the commit if a hand-edited `versionId` does not equal the derived value. Materializing it at pre-commit (not build-only) means downstream consumers read a stable on-disk field rather than re-deriving it.

The generator must read `version` **as a string**, not a YAML-coerced scalar: `version: 1.0` parses as the float `1.0` and would mint `nist-ai-rmf@1` (the trailing zero lost), so `version` is authored quoted (`'1.0'`) and the generator asserts string type before composing. The generator additionally asserts each composed `versionId` matches a conservative charset (`[a-z0-9.@-]`, no whitespace) and that the set of `versionId`s across the registry is unique — two entries minting the same `versionId` is a registry error the generator fails on. The same well-formedness applies to the lineage fields (D2c, M4): every member of `supersedes`/`priorVersions` must itself be a charset-valid `versionId`, unique within its list, and belong to the **same concept-id family** as the entry carrying it (`nist-ai-rmf@1.0` may appear in lineage on the `nist-ai-rmf` entry, never on `mitre-atlas`). The Tier-1 version-known join (D5) is a literal match of a pinned value's token against this set, so a malformed or cross-family lineage member would silently break the join — the generator fails the commit on it.

`versionId` serves three roles: the human-readable citation surface ("mapped against MITRE ATLAS 5.0.1"), the stable target a supersession chain (D2c) points at, and the **token vocabulary the pinned mapping value (D3) draws from** — a mapping pinned to `mitre-atlas@5.0.1` is exactly a value bound to that `versionId`. It is **not** introduced into the `mappings` keys (D7).

#### D2c. Supersession lineage

Two optional registry-internal fields, modeled on DataCite `RelatedIdentifiers`, record version lineage so a reader sees the family history without external lookup:

- `supersedes` — the prior `versionId` this version replaces (`nist-ai-rmf@1.0` on the entry that becomes `nist-ai-rmf@1.1`). Optional; absent on first registration.
- `priorVersions` — an optional ordered list of retired `versionId`s for the family that CoSAI **still recognizes as valid pin targets**.

These are *registry* lineage, not resolution: they do not move any mapping. A mapping pinned to `nist-ai-rmf@1.0` stays pinned to `1.0` after the registry advances to `1.1`; the lineage fields let the drift validator (D5) and a human reader understand the relationship between the pinned version and the current one.

`priorVersions` carries a specific semantic that the drift validator depends on (D5/D5a, M3): **membership in `priorVersions` marks a pin as `valid-but-superseded` — a valid pin target, sanctioned, NOT a validation failure.** The two halves matter independently. The "valid" half: a `priorVersions`-listed pin passes validation (no CI break, no schema failure, no `invalid` flag); deliberately keeping an old-edition mapping is a first-class state, in line with multi-version coexistence (D3). The "superseded" half: the drift validator (D5a) still classifies it *distinctly* from a pin on the current `version` and reports it informationally, so a maintainer can audit which mappings live on prior editions without those mappings being treated as broken. The full three-state taxonomy (`current` / `valid-but-superseded` / `invalid`) is in D5a; a pin to a version that is *neither* current *nor* in `priorVersions` is the `invalid` case the drift validator fails on — the forgotten/un-reviewed pin. The version-change process (D10) is where a maintainer decides, per superseded mapping, whether to re-pin it forward (move it back to `current`) or sanction the retention by leaving the old version in `priorVersions` (`valid-but-superseded`).

### D3. Per-mapping version pinning in the value — the integrity mechanism

A mapping value carries the framework version it was authored against, **pinned in the value**, generalizing OWASP's existing `LLM02:2025` shape across all versioned frameworks. Concretely, a versioned mapping value is the **spec-native canonical reference plus a version token**, where the version token is the same `@<version>` suffix the `versionId` (D2a) uses:

| Framework | Spec-native ref | Pinned value (target form) |
|---|---|---|
| `mitre-atlas` | `AML.T0043` | `AML.T0043@5.0.1` |
| `nist-ai-rmf` | `GOVERN-6.2` | `GOVERN-6.2@1.0` |
| `eu-ai-act` | `Article 50` | `Article 50@2024` |
| `owasp-top10-llm` | `LLM02` | `LLM02:2025` (year-in-value retained — see D6) |
| `stride` | `Tampering` | `Tampering` (unversioned — `version: null`, no token) |
| `iso-22989` | `AI Partner (data supplier)` | `AI Partner (data supplier)@2022` (controlled-vocab enum + token — D8) |
| `nist-ai-rmf-actor-tasks` | `AI Deployment` | `AI Deployment@1.0` (controlled-vocab enum + token — D9) |
| `etsi-en-304-223` | `End-users` | `End-users@2.1.1` (controlled-vocab enum + token — D11) |

**Why the value, not the key (the OWASP generalization).** OWASP is the proof case: its value already self-pins (`LLM02:2025`), so a 2023 mapping and a 2025 mapping coexist as distinct values under the *same* `owasp-top10-llm` key, and a validator can tell them apart. We generalize that property to every versioned framework. Putting the version in the *value* — never the key — means: the `framework.id` enum stays a closed key set (no key explosion per version); the D5b *key* set is byte-stable; the `propertyNames` `$ref` consumers share is untouched (D7). The cost lands entirely in the D5b *value* patterns (they grow a version dimension) and in the content values (they migrate to pinned form) — that is #343 (D7).

**The version token's delimiter is framework-determined, not uniform.** Every framework except OWASP uses the `@<version>` token, giving its mapping value's version token one-to-one correspondence with its `versionId` token (`AML.T0043@5.0.1` ↔ `mitre-atlas@5.0.1`). OWASP keeps its `:YYYY` token: `:` is OWASP's own upstream-recognizable convention (the 2025 edition is literally titled with the year), it is already the committed D5b pattern, and forcing it to `@` would churn the one framework that is already integrity-safe for no gain. This is a deliberate split, and it has a parse consequence #343 consumers must respect: the value-token delimiter (`@` for most, `:` for OWASP) does **not** match the `versionId` delimiter for OWASP (value `LLM02:2025` vs `versionId` `owasp-top10-llm@2025`). A consumer that splits a pinned value to recover its version must select the delimiter **per framework** (driven by the framework `id`), never assume a single character across all frameworks. STRIDE carries no token at all (`version: null`).

**What pinning buys (replacing latest-wins).** Per-value pinning replaces registry latest-wins with explicit per-mapping resolution. What this delivers *at adoption* is that re-versioning no longer silently re-points an existing mapping, and that drift becomes expressible (D5 Tier 1 can check every pin against the known version set and the controlled vocabulary). What it makes *possible later* (D5 Tier 2, once per-version catalogs exist) is detecting that a pinned ref is absent from a newer edition. Concretely, against the three failure shapes:

- *Semantic shift (OWASP).* A 2023 `LLM02:2023` and a 2025 `LLM02:2025` are different values; advancing the registry to 2025 does not silently re-point the 2023 mapping. They coexist during a transition and are individually citable. **Caught at adoption** as a side effect of the values being distinct — the re-interpretation cannot happen silently.
- *Renumber (EU AI Act, prospective).* `Article 52@2021` and `Article 50@2024` would be distinct values, both valid against their pinned editions; neither silently re-resolves. **The silent-re-resolution is prevented at adoption** by the pin. Detecting that `Article 52` no longer exists in the 2024 text is a Tier-2 catalog check, not an adoption-time guarantee. (EU AI Act has no mapping content today; this is prospective via #164.)
- *Retire/dangle (ATLAS).* `AML.T0043@5.0.1` records the matrix it was authored against. Tier 1 can confirm `5.0.1` is a version CoSAI recognizes. Flagging that `AML.T0043` is *absent from a later matrix* — the actual dangling detection — requires a per-version ATLAS technique catalog and is **Tier 2, deferred**. Pinning makes this detectable later; it is not detected at adoption.

Multi-version coexistence is first-class: a content file may legitimately hold both `AML.T0043@5.0.1` and `AML.T0043@6.0.0` during a migration window, which a concept-id-stable latest-wins model cannot express.

**What Tier 1 does and does not check (M2).** The version token being registry-known is *not* a claim that the ID-bearing ref is correct *for* that edition. `Article 52@2024` — a pin whose `2024` token is recognized but whose `Article 52` is the wrong (proposal-era) article number for the 2024 final text — passes the schema pattern and Tier 1, because Tier 1 has no per-version article catalog to check the ref against. Catching a ref that is wrong-for-its-pinned-version is Tier-2 (catalog-absence, D5), deferred. The token grammar guarantees *version-known*, not *ref-correct-for-version*.

#### D3a. Version-token grammar — the normalized `version` string, schema-anchored

The token after the framework-determined delimiter is **the normalized `version` string verbatim** — byte-for-byte the same string the registry `version` field holds for that edition (`5.0.1`, `1.0`, `2025`, `2.1.1`). It is not a re-formatting, an abbreviation, or a free-form annotation: `AML.T0043@5.0.1` carries exactly the `version: '5.0.1'` token, and OWASP's `LLM02:2025` carries exactly the `version: '2025'` token after its `:` delimiter. This makes the value-token ↔ `versionId`-token correspondence (D2a) exact for every framework except OWASP's delimiter choice, and makes the join in D5 Tier 1 a literal string match against the registry's recognized version set.

The schema's pinned-value sub-pattern is therefore **anchored to that known version set** — an alternation over the framework's current `version` plus its `priorVersions` members — not a loose `.+` / `@[\w.]+` that would admit any token. The anchored alternation is what lets `check-jsonschema` enforce *version-known* at the schema layer for the ID-bearing frameworks; a loose token would push the entire version-known check onto the Tier-1 drift validator (D5) and let an unknown-version pin pass schema validation. The alternation members are regenerated alongside the `versionId`s (D2b) and widen when D10 advances the recognized set (`@(5\.0\.1)` → `@(5\.0\.1|6\.0\.0)`).

This grammar rests on a load-bearing parse invariant (H3): **the `@` delimiter — and OWASP's `:` delimiter — never appears in any controlled-vocabulary base value, in any spec-native ID-bearing ref, or in any concept `id`.** Concept ids are the closed kebab enum (no `@`, no `:`); ID-bearing refs are dotted/dashed/`Article N(n)` forms (no `@`); and the controlled-vocabulary base values, though they contain spaces and parentheses (`AI Partner (data supplier)`, `End-users`), contain neither `@` nor `:`. So splitting a pinned value on its (framework-determined) delimiter cleanly recovers `(base-ref, version)` even for the space- and paren-bearing non-ID values — the delimiter is the unique last `@` (or, for OWASP, the unique `:`). A consumer must still select the delimiter **per framework** (driven by the framework `id`), never assume one character across all frameworks (D6/M1).

### D4. Mapping-value generation — `framework-mapping-maintainer.py`, derived `mappingId`, purity-validated

Per-mapping pinning is **generated, not authored**, under the same generate-not-author principle as the registry `versionId` (D2b). A contributor — or a new-mapping issue — supplies only a *structured selection*: the CoSAI entity, the framework, the framework version, and the framework-specific reference. A maintainer/dev tool emits the pinned value and writes the YAML. The contributor never types the pinned token or the version string.

#### D4a. CLI shape

```
framework-mapping-maintainer.py [add|update|remove] \
  --cosai-id <riskXXX|controlXXX|componentXXX|personaXXX>   # CoSAI entity (camelCase impl id) \
  --framework <framework id from frameworks.yaml> \
  --version   <version from that framework's entry> \
  --framework-specific-ref <GOVERN-1.1 | AML.T0043 | Article 50 | AI Partner (data supplier) | ...>
```

`add`, `update`, and `remove` all address a mapping **positionally** by the same tuple — `--cosai-id` + `--framework` + `--version` + `--framework-specific-ref` — not by a single opaque handle (D4b). This is deliberate: the pinned values legitimately contain spaces and parentheses (`AI Partner (data supplier)@2022`, `Article 50@2024`), so a single `--mappingId <id>` literal would force a contributor to type a space- and paren-bearing string and force an escaping convention for it. Positional addressing sidesteps that: the tool composes the pinned value from the structured tuple for `add`, and for `update`/`remove` it resolves the same tuple to the exact existing list entry. No space- or paren-bearing handle is ever typed on the command line.

The tool: (1) validates `--framework` against the `frameworks.yaml` `id` enum; (2) validates `--version` against that framework's `version` field (and, once `priorVersions` is populated, against the recognized version set) — a `--version` the registry does not know is rejected at authoring time, the earliest possible point; (3) validates `--framework-specific-ref` against the framework's value constraint for that version — the spec-native D5b regex for ID-bearing frameworks (the base pattern, before the version token), or the **closed controlled-vocabulary set** (D8/D9/D11) for the non-ID-bearing frameworks (a `--framework-specific-ref` not in the closed set is rejected, so `AI Part (Data supplier)` cannot be authored); (4) composes the pinned value (`<ref>@<version>`, or `<ref>:<year>` for OWASP, or bare `<ref>` for unversioned STRIDE); (5) writes the mapping into the target entity's `mappings.<framework>` block. This is the same generated-artifact contract as templates, tables, and `versionId`: tooling writes, a purity check proves no hand-tampering.

#### D4b. Positional addressing, derived (non-stored) `mappingId`

The CLI's `update`/`remove` verbs require addressing an individual mapping. Addressing is **positional by the structured tuple** (`--cosai-id` + `--framework` + `--version` + `--framework-specific-ref`, D4a), so no opaque space- or paren-bearing handle is ever typed. The internal `mappingId` the tool uses to key a located entry is **derived, not stored**, composed deterministically from that same tuple. To keep the derived handle token-safe despite the spaces and parentheses in the pinned values, composition is **binding** as: join the tuple components with a reserved `|` separator into the canonical string `<cosai-id>|<framework-id>|<pinned-value>`, then take a stable hash of that string (a hex digest) as the handle. The hash is what makes the handle token-safe and collision-resistant regardless of the spaces/parens/`@`/`:` in the pinned value; the `|`-joined canonical string is well-defined because the parse invariant (D3a) guarantees `cosai-id` and `framework-id` are charset-clean and the pinned value's internal structure is recoverable without `|`. The `mappingId` is never written to YAML.

Mapping values therefore **stay bare-string arrays** (`mappings.<framework>: [<pinned>, ...]`), exactly the shape every consumer assumes today. The consumer schemas (`risks`/`controls`/`components`/`personas`) are **unchanged**: value-pinning does not reach beyond `frameworks.schema.json` (D7).

This is a binding decision, not a deferred one. The alternative — an explicit stored `mappingId` field carried alongside each value — would make `mappingId` stable across value edits, but at the cost of changing the value shape in all four consumer schemas, which the derived form avoids entirely. The only property the derived form gives up is stability of the handle across a value edit (changing a pinned value changes its derived `mappingId`), which is acceptable because the tool addresses mappings within a single invocation against the on-disk state, not across edits. Should stable-across-edits addressing ever be required, introducing a stored field is a **future amendment to this ADR**, not implementation latitude for #343.

#### D4c. Purity validator

A validator (paired with the generator, same class as the `versionId` purity check in D2b) re-derives each managed mapping value from its structured inputs and fails the commit on mismatch — proving pinned values were generated, not hand-edited. For frameworks the tool manages, a hand-typed `AML.T0043` (un-pinned) or `AML.T0043@9.9.9` (unknown version) fails this check.

### D5. Drift-detection validator — what ships now, what comes later

A new **drift-detection validator** consumes the pinned values. Its capability splits into two tiers, and the split is load-bearing for the honest integrity claim: **only Tier 1 ships at adoption**, and Tier 1 does *not* detect technique-retirement or article-renumber — it detects version-not-known and vocabulary-non-membership.

- **Tier 1 (version-known check + controlled-vocabulary membership) — ships with the pinning migration.** For every pinned mapping value, Tier 1 checks two things, using only data already on disk (no external catalog):
  1. **Version known.** The value's version token must be a version the registry recognizes for that framework — the current `version` *or* a member of `priorVersions` (D2c). A token outside that set (a typo, or a pin to a version CoSAI never recorded) is flagged.
  2. **Controlled-vocabulary membership (non-ID frameworks).** For ISO 22989 / NIST Actor Tasks / ETSI (D8/D9/D11), the base ref must be a member of the closed inline `enum` for the pinned edition. A value outside the closed set (`AI Part (Data supplier)`) is flagged even though it is a syntactically valid string. For the ID-bearing frameworks this membership role is already played by the spec-native D5b regex enforced by `check-jsonschema`.

#### D5a. Classification — the three states Tier 1 produces

For each pinned mapping value, Tier 1 classifies the result into one of three states, computed entirely from `frameworks.yaml` (the entry's current `version` and its `priorVersions` list — no other inputs):

| State | Definition | Validator behavior |
|---|---|---|
| `current` | The pinned version token equals the framework's current `version` (or the framework is unversioned, e.g. STRIDE — no token to compare). | **Pass silently.** |
| `valid-but-superseded` | The pinned version token is listed in `priorVersions` — a sanctioned retention from a prior edition (D2c). | **Pass + report informationally** (audit surface, not a CI failure). |
| `invalid` | The pinned version token appears in *neither* the current `version` *nor* `priorVersions` — a typo, stale copy, or pin to a version CoSAI never recorded. | **Fail.** |

The taxonomy resolves what could otherwise read as a contradiction between D2c (`priorVersions` makes pins **valid**) and D10 (drift process **surfaces** old pins for review): both are true at the same time — a `valid-but-superseded` pin is *valid* (not failure-worthy) **and** *reported* (audit surface). Three contributor/maintainer workflows it supports concretely: **(a)** a contributor deliberately pins to an older edition during a transition window — the validator passes, no PR break; **(b)** a maintainer runs the drift validator to get an informational audit report of every `valid-but-superseded` pin, decides per-mapping whether to re-pin forward or leave retained (D10b step 4); **(c)** a typo or stale copy producing an unknown version → `invalid`, fails. The same three states are what the controlled-vocabulary (D8/D9/D11) membership check produces for the non-ID frameworks: a base ref in the closed enum for the pinned edition is `current`/`valid-but-superseded` (depending on the version token), and a base ref outside the closed set is `invalid` — the version axis and the vocabulary axis classify together.

- **Tier 2 (catalog-absence check) — deferred, depends on per-version catalogs CoSAI does not yet hold.** When per-version framework catalogs become available (e.g. an ATLAS technique list per matrix version, or the EU AI Act article set per edition), Tier 2 flags a pinned `<ref>@<version>` whose `<ref>` is absent from that version's catalog — the actual **retire/renumber/dangle detection**. This is the tier that would catch a retired ATLAS technique or a renumbered EU article. It requires importing or caching per-version catalogs and is explicitly a follow-up, not a prerequisite for adopting pinning, and **not delivered at adoption**.

The honest scope of the immediate guarantee: pinning plus Tier 1 makes drift *expressible and version-checkable now*, and makes catalog-level dangling detection *possible later* (Tier 2). It does not detect at adoption that a referenced technique was retired or an article renumbered. `validate_framework_references.py` stays responsible for registry existence and `applicableTo`; the drift validator is a separate concern.

### D6. Mapping-convention harmonization — spec-native canonical forms plus a version token

We **reject** the CoSAI-invented structured form (`<framework>:<version>:<axis>:<sub-classifier>`, e.g. `iso22989:2022:ai_customer:ai_user`) proposed in #153, for all frameworks. We **keep each framework's spec-native canonical form** as already committed in D5b and harmonize by migrating the legacy content forms up to those canonical forms, **then append the version token** (D3). D6 governs the **ID-bearing** frameworks (MITRE ATLAS, NIST AI RMF Subcategories, EU AI Act, OWASP, STRIDE), whose spec-native regex already constrains the value space; the **non-ID-bearing** frameworks (ISO 22989, NIST Actor Tasks, ETSI) get the complementary controlled-vocabulary treatment in D8/D9/D11 — closed enums in place of free strings — which is the half of #153's structured-form instinct that is correct (validate against a closed set) without the cross-framework dialect (the half D6 rejects). The decisive reasons against the structured form: it invents a CoSAI dialect contributors must learn, diverges from every upstream's own identifier practice, and loses the spec-native readability the style guide names as the primary purpose of mappings. Per-value version pinning achieves the *integrity* the structured form was reaching for (version visible in the value) **without** the cross-framework dialect — the version rides as a suffix token on the otherwise spec-native ref.

STRIDE is a special case worth stating explicitly. It is `version: null`, so its pinned values carry **no version token** (`Tampering`, not `Tampering@...`) — it appears exempt from the integrity mechanism, but it is not. STRIDE's six-value set is a **closed enum frozen by the D5b schema** (`Spoofing`, `Tampering`, `Repudiation`, `InformationDisclosure`, `DenialOfService`, `ElevationOfPrivilege`), and that closed enum *is* STRIDE's integrity guarantee: a value outside the set is rejected by `check-jsonschema`, and the set cannot change without a schema edit. STRIDE is therefore **pinned by enum rather than by token** — its integrity comes from a frozen membership set, and any change to that set is a schema edit gated by the version-change process (D10), exactly as the controlled-vocabulary enums (D8/D9/D11) are. The absence of a token reflects that STRIDE does not re-version, not that it is unguarded. To be precise about what "frozen" governs (M3): the *membership set* (the six PascalCase values) is frozen by the schema, while the *legacy content spelling* (lowercase-kebab `tampering`, `information-disclosure`) migrates **to** that frozen set as part of #343 — "frozen enum" and "STRIDE values migrate form" are not in tension, because the migration normalizes existing values into the frozen set's canonical spelling, it does not change the set.

The canonical *base* forms (before the version token) are:

| Framework | Legacy in content (verified) | Canonical base (D5b) | Pinned target (D3) |
|---|---|---|---|
| `nist-ai-rmf` | `GV-6.2`, `MS-2.3`, `MP-3.4`, `MG-2.1` | `GOVERN-6.2`, `MEASURE-2.3`, `MAP-3.4`, `MANAGE-2.1` | `GOVERN-6.2@1.0` |
| `stride` | `tampering`, `information-disclosure` | `Tampering`, `InformationDisclosure` | `Tampering` (unversioned) |
| `owasp-top10-llm` | `LLM04` | `LLM04` + edition | `LLM04:2025` |
| `mitre-atlas` | `AML.T0020` (canonical base) | unchanged base | `AML.T0020@5.0.1` |
| `eu-ai-act` | `Article 3(3)` (canonical base) | unchanged base | `Article 3(3)@2024` |
| `iso-22989` | `AI Partner (data supplier)` (bare) | closed controlled-vocab enum (D8) | `AI Partner (data supplier)@2022` |

**OWASP year-in-value divergence — retained as the prototype.** The D5b `owasp-top10-llm` pattern `^LLM\d{2}:\d{4}$` embeds the year in the value (`LLM01:2025`), which diverges from OWASP's own practice of keeping entry IDs stable (`LLM01`) and putting the year in a separate field. That divergence is precisely what makes OWASP integrity-safe by construction, and it is the prototype this ADR generalizes: year-in-value is per-mapping pinning, already shipped for one framework. We retain `LLM01:2025` deliberately, and accept the **two-token-style split** (OWASP uses `:YYYY`, all other versioned frameworks use `@<version>`) as a named, documented divergence — OWASP's `:` form is upstream-recognizable and already committed; harmonizing it to `@` would churn the one already-correct framework. The style guide must teach both token styles and explain why, and (M1) consumers parsing a pinned value to recover its version must pick the delimiter by framework.

### D7. ADR-022 D5b / ADR-016 D3 collision impact analysis

This is the collision analysis against the committed ADR-022 D5b and ADR-016 D3 surfaces under the value-pinning posture.

**Verdict: the mapping keys, the `framework.id` enum, the `propertyNames` `$ref`, and the four consumer schemas (`risks`/`controls`/`components`/`personas`) stay byte-stable — there is no key collision and no consumer-schema change. What changes is confined to `frameworks.schema.json`'s `framework-mapping-patterns` *value* entries — the ID-bearing patterns grow a version dimension and the non-ID-bearing entries change from `{"type": "string"}` to a closed, version-pinned inline `enum` (D8/D9/D11) — plus the content mapping values, which migrate to pinned form. The `framework.id` enum gains exactly two new keys, `nist-ai-rmf-actor-tasks` (D9) and `etsi-en-304-223` (D11). ADR-016 D3 is untouched.**

**What does NOT change (no key collision, no consumer-schema change).** The collision the trio feared was the *version-in-key* posture (`mappings.mitre-atlas-v5: [...]`), which would have forced (a) the D5b `framework-mapping-patterns` object *keys* to grow per version, (b) the `framework.id` enum and the shared `propertyNames` `$ref` to admit suffixed keys, and (c) a re-key of every content block. Value-pinning puts the version in the *value*, so none of (a)/(b)/(c) occurs: the mapping keys stay the closed concept-id set, the enum changes only by the two genuinely new frameworks, and `propertyNames` is untouched. The consumer schemas are equally untouched: because `mappingId` is **derived, not stored** (D4b, binding), mapping values stay bare-string arrays, so `risks`/`controls`/`components`/`personas` carry the same `mappings.<id>: [...]` array shape they carry today. Value-pinning does not reach beyond `frameworks.schema.json`. This is the whole reason pinning goes in the value, and addressing goes in a derived id.

**What DOES change (the value patterns).** Value-pinning changes the D5b value patterns. Each versioned framework's pattern must admit the version token:

| Framework | D5b pattern today | D5b pattern under pinning (illustrative) |
|---|---|---|
| `mitre-atlas` | `^AML\.(T\|M)\d{4}(\.\d{3})?$` | `^AML\.(T\|M)\d{4}(\.\d{3})?@(5\.0\.1)$` |
| `nist-ai-rmf` | `^(GOVERN\|MAP\|MEASURE\|MANAGE)-\d+(\.\d+)*$` | `^(GOVERN\|MAP\|MEASURE\|MANAGE)-\d+(\.\d+)*@(1\.0)$` |
| `owasp-top10-llm` | `^LLM\d{2}:\d{4}$` | unchanged (already version-bearing) |
| `eu-ai-act` | `^Article\s\d+(\(\d+\))?$` | `^Article\s\d+(\(\d+\))?@\d{4}$` |
| `stride` | `^(Spoofing\|...\|ElevationOfPrivilege)$` | unchanged (unversioned, `version: null`) |
| `iso-22989` | `{"type": "string"}` (no constraint) | closed `enum`, `@<version>`-pinned (D8) |
| `nist-ai-rmf-actor-tasks` *(new key, D9)* | (did not exist) | closed `enum`, `@<version>`-pinned |
| `etsi-en-304-223` *(new key, D11)* | (did not exist) | closed `enum`, `@<version>`-pinned |

The version-token sub-pattern is **bound here, not left to #343** (D3a): it is an alternation anchored to the framework's known version set (the current `version` plus any `priorVersions` members), not a loose `.+` / `@[\w.]+`. This is what lets `check-jsonschema` enforce "version-known" at the schema layer for the ID-bearing frameworks — without the anchored alternation, the schema would admit any token and the entire version-known burden would fall on the Tier-1 drift validator (D5). The exact alternation members track the registry's recognized version set and are regenerated when D10 advances it; the illustrative single-member alternations above (`@(5\.0\.1)`, `@(1\.0)`) widen to `@(5\.0\.1|6\.0\.0)` as `priorVersions` grows. The architectural commitments are: **the four versioned ID-bearing patterns (ATLAS, NIST Subcategories, EU AI Act, plus OWASP which is already version-bearing) carry the anchored version token**, STRIDE stays unversioned, and the **three non-ID-bearing entries (ISO 22989, NIST Actor Tasks, ETSI) change from a free `{"type": "string"}` to a closed `enum`** of version-pinned values (D8/D9/D11). These are *value*-pattern edits within `frameworks.schema.json`'s `framework-mapping-patterns`; the *keys* of that object grow only by the two genuinely new frameworks (`nist-ai-rmf-actor-tasks`, `etsi-en-304-223`), each mirrored into the `framework.id` enum.

**The controlled-vocabulary mechanism (D8/D9/D11) and how it versions.** A non-ID framework's value space for a given edition is a closed list published in the spec's own role/task table. We encode it as an **inline per-version `enum` in `framework-mapping-patterns`**: the entry becomes a `oneOf`/`anyOf` over `enum`s, each enum carrying its version-suffixed members (e.g. ISO 22989 `["AI Partner (data supplier)@2022", "AI Customer@2022", ...]`). This is the **binding encoding** for the non-ID frameworks, chosen over a slug-plus-registry mechanism (see Alternatives Considered) because the sets are small and stable (ISO 22989 roles, NIST Actor Tasks, ETSI's five roles), `check-jsonschema` enforces an inline `enum` directly with no extra validator, and self-containment in the schema is simplest to drift-test. The vocabulary is **per-framework-version**: a new edition gets its own enumerated set keyed to its version token, and old `@<version>` pins continue to validate against the edition they were authored against — the same integrity property D3 gives the ID-bearing frameworks. The drift validator's Tier 1 membership check (D5) and the version-change process's vocabulary refresh (D10) both operate against this inline-enum encoding.

**Consumer-schema reach — none.** Value-pinning does not touch `risks`/`controls`/`components`/`personas` schemas: pinned values are still strings in the same `mappings.<id>: [...]` arrays, and addressing is positional with a derived (non-stored) `mappingId` locator (D4b, binding), so no consumer schema changes. This is unconditional under the decisions above, not contingent on a later choice.

**Content migration.** Every populated mapping value migrates from un-pinned to pinned form (`AML.T0020` → `AML.T0020@5.0.1`, `GV-6.2` → `GOVERN-6.2@1.0`, `LLM04` → `LLM04:2025`, etc.), folding the legacy-form harmonization (D6) and the version-pinning into one rewrite. The migrated unit is the **individual mapping value**: 46 entities carry mappings across the three populated files, spread over 96 `mappings.<framework>:` blocks (risks 65, controls 25, personas 6), comprising **146 individual values** (risks 100, controls 40, personas 6). This is **#343 execution**, performed by the `framework-mapping-maintainer.py` tool (D4) plus a one-time migration pass, then the consumer-schema catch-alls flip to strict so `check-jsonschema` enforces the version-bearing patterns. STRIDE values migrate form only (no token).

**Cross-branch ordering invariant (M1).** The strict pattern lives in `frameworks.schema.json` (a schema/infrastructure change, base `main`), but the values it strict-validates live in content (`risks`/`controls`/`personas`, base `develop`). These travel on different branches with `develop` periodically bridged to `main`, so the migrated content and the strict pattern do not land atomically. The ordering invariant is therefore binding: **the migrated values must reach the same branch as the strict pattern before the catch-all is removed.** Concretely, the schema keeps the loose catch-all (or a warn-only posture) until the migrated content has bridged from `develop` into `main`; only then does the catch-all flip to block. This mirrors the C2 warn-then-block linter precedent and prevents a strict-flip on `main` from rejecting still-legacy content that is correct on `develop` but awaiting the develop→main bridge. **EU AI Act is prospective, not part of this migration**: it has zero mapping content on disk today and enters the corpus through the unlanded #164, so it adopts the pinned form at first authoring rather than being migrated.

**ADR-016 D3 (`externalReferences[].id` canonical forms) is not touched.** D3's `atlas-AML.TNNNN`-style external-reference ids are a different surface (per-entry citations, not framework cross-walk values), and nothing here changes them. They remain version-blind, which is correct for a citation to a stable technique page.

**Summary of the cost.** Integrity and non-ID standardization cost exactly: the four versioned ID-bearing value patterns gain a token, the three non-ID-bearing entries become closed version-pinned inline enums, the `framework.id` enum gains two new keys (`nist-ai-rmf-actor-tasks`, `etsi-en-304-223`), and the 146 content values migrate to pinned form (#343) — with the mapping *keys* of existing frameworks, `propertyNames`, the four consumer schemas, and ADR-016 untouched. The cost is paid in the value layer, by tooling, not by contributors.

### D8. ISO 22989 — controlled-vocabulary carve-out plus version token

ISO 22989 persona mappings are role descriptors (`AI Partner (data supplier)`), not canonical catalog IDs, so no spec-native regex constrains them. A free string is wrong for a standard: `AI Partner (data supplier)`, `AI Partner (datasupplier)`, and `AI Part (Data supplier)` all validate and nothing flags the divergence. ISO 22989's roles are a closed published set, so we constrain them with a **controlled vocabulary** — the closed set of ISO/IEC 22989:2022 AI roles and sub-roles drawn from the standard's own role taxonomy — encoded as a closed, version-pinned inline `enum` (D7). A value outside that closed set is **rejected** by `check-jsonschema` directly, and the drift validator's Tier 1 membership check (D5) covers the same set as a belt-and-suspenders join.

The roles on disk today are six forms — `AI Producer`, `AI Partner (data supplier)`, `AI Partner (infrastructure provider)`, `AI Partner (tooling provider)`, `AI Customer (application builder)`, `AI Customer (end user)` — and the closed enum for the `2022` edition is sourced from the standard's own role taxonomy (final set owned by #319/#329 provenance, not fixed here). The vocabulary is **per-framework-version**: ISO 22989 *is* versioned (`version: '2022'`, amendments expected), so each role carries the `@<version>` token (D3) and the pinned form is `AI Partner (data supplier)@2022`. The closed set is keyed to the `2022` edition; when ISO 22989:2022 is amended the new edition gets its own enumerated set and existing `@2022` pins continue to validate against the edition they were authored against — they do not silently re-resolve. The generator (D4) composes the pinned value from the contributor's role selection; the drift validator (D5 Tier 1) checks both the `@2022` token and closed-set membership against the registry. The architectural commitment is a closed set, not a free string.

### D9. NIST AI RMF persona axis (#319) — Actor Tasks as a separate framework `id`

We admit NIST AI RMF Actor Tasks as a **separate framework `id`**, `nist-ai-rmf-actor-tasks`, distinct from the existing `nist-ai-rmf` (Subcategory) framework (#319 Option C). The existing `nist-ai-rmf` entry keeps its `^(GOVERN|MAP|MEASURE|MANAGE)-\d+(\.\d+)*$` Subcategory base pattern (now `@`-token-bearing per D7) unchanged and stays the control-surface mapping used by `controls.yaml`. The new `nist-ai-rmf-actor-tasks` entry carries the lifecycle-role axis (`applicableTo: [personas]`, value space = the text-only Actor-Task names from AI RMF 1.0 Appendix 3, including the "Additional AI Actors" End Users / Third-party Entities raymondsheh flagged).

Rejecting the alternatives: re-purposing the `nist-ai-rmf` pattern to free-form (Option B) would break the existing Subcategory mappings on controls; a structured discriminator within one id (`TASK:` / `SUBCAT:`, Option D) would migrate every existing `GOVERN-X.Y` to `SUBCAT:GOVERN-X.Y` and couple to the structured form D6 rejected. Two ids from one source spec is unusual but architecturally honest: Subcategories and Actor Tasks are genuinely two axes of one document (normative-control surface vs lifecycle-role descriptor), each matching a different CoSAI surface (controls/risks vs personas).

**Value-shape for Actor Tasks:** because AI RMF assigns no canonical IDs to Actor Tasks, `nist-ai-rmf-actor-tasks` is a **controlled-vocabulary carve-out** in the same posture as ISO 22989 (D8) — text-only task names, but a *closed* set, not a free string. The closed set is the Actor-Task names enumerated in AI RMF 1.0 Appendix 3 (AI Deployment, Operation and Monitoring, TEVV, Procurement, Governance and Oversight) plus the "Additional AI Actors" raymondsheh flagged (End Users, Third-party Entities), encoded as a closed, version-pinned `enum` per D7 with the `@<version>` token (`AI Deployment@1.0`). A task name outside the closed set is rejected. Adding the framework is a coordinated edit per ADR-022's "new framework" follow-up: `frameworks.yaml` entry (with a generated `versionId` of `nist-ai-rmf-actor-tasks@1.0`), `frameworks.schema.json` `id`-enum extension, a `framework-mapping-patterns` entry (closed `enum` + token), and `applicableTo: [personas]`. **This is content/schema work owned by #319 execution and #343, not performed here** — including pinning the exact enumerated Appendix-3 task list.

### D10. Version-change process — what a framework re-issuance triggers

The decisions above model the version *data*; D10 records the *workflow* for when an external framework publishes a new edition. A version bump is **a gated, owned operation, not an ad-hoc edit.** It is not automatic: nothing in CoSAI advances a framework's `version` until a maintainer runs the process, precisely so that no existing pinned mapping silently re-resolves (the inverse of the latest-wins failure D3 closes).

#### D10a. Trigger and gate, scaled to blast radius

A new published edition (an ETSI EN re-issue, a NIST AI RMF 1.1, an ISO amendment, an ATLAS quarterly matrix) is the trigger. The gate's weight scales to **how many existing pinned values the bump affects** — determined by running the D5 drift sweep before the registry `version` flips:

- **Lightweight path (zero affected pins).** If the drift sweep shows no content pinned to the version being superseded — common for high-cadence frameworks like ATLAS quarterly, where CoSAI may have authored nothing against the prior matrix — the bump is a reviewed-but-lightweight registry edit: advance `version`, regenerate `versionId`, record lineage, and (for non-ID frameworks) add the new edition's enum. No per-mapping migration is needed because nothing is affected. This does not require a dedicated tracking issue with the full checklist; an ordinary reviewed change suffices.
- **Full path (one or more affected pins).** If the drift sweep flags content pinned to the superseded version, the bump **requires a tracking issue carrying the full D10b checklist** before the registry `version` flips, because the bump now entails per-mapping migration decisions that need review.

In both paths the `frameworks.yaml` `version` field and the controlled-vocabulary enum (D8/D9/D11) advance only under review, never via an unreviewed in-place edit; what scales is the ceremony, not the requirement for review.

#### D10b. The full bump checklist (full path only)

When the drift sweep flags affected pins, the version-bump issue drives, in order:

1. **Registry update** — add the new edition to the framework entry: advance `version`, regenerate `versionId` (D2b), and record lineage (`supersedes` ← prior `versionId`, append prior to `priorVersions`, D2c). The prior version stays recognized (in `priorVersions`) so existing pins remain valid and un-flagged (D5/M3).
2. **Controlled-vocabulary refresh (non-ID frameworks)** — add the new edition's closed inline enum (D7) for ISO 22989 / NIST Actor Tasks / ETSI, sourced from the new edition's role/task table. The old edition's enum is retained (per-version, D7), so `@<old>` pins still validate.
3. **Drift sweep** — run the D5 drift validator across all content. Tier 1 classifies each pinned mapping into one of the three states (D5a): `current`, `valid-but-superseded`, or `invalid`. Any `invalid` pins (pinned to a version that is neither current nor in `priorVersions`) **fail** and must be resolved before the bump. `valid-but-superseded` pins are informational — they are the audit surface the per-mapping migration decision (step 4) acts on, never a failure. Tier 2 (when catalogs exist) surfaces refs absent from the new edition.
4. **Content migration decision (per mapping)** — for each flagged mapping a maintainer decides *re-pin* (the ref still means the same thing in the new edition — re-author via the D4 generator to the new token) or *retain* (the mapping is deliberately about the old edition — leave the old pin and ensure the old version is listed in `priorVersions` so D5 treats it as sanctioned, not stale). This is a content decision, never an automatic rewrite; multi-version coexistence (D3) makes "retain" first-class.
5. **Re-doc** — update `framework-mappings-style-guide.md` if the new edition changes the canonical/enumerated forms.

**Exit gate:** the issue closes only when the drift validator reports no `invalid` pins (D5a) — any intentional retentions have been sanctioned by recording the old version in `priorVersions` (D2c) so they classify as `valid-but-superseded`, not `invalid` — and the style guide reflects the new edition.

#### D10c. Ownership and cadence

Framework version metadata is **maintainer-owned** (the same surface that owns `frameworks.yaml` and the schema). Cadence is **on-publish / on-demand, not scheduled**: the trigger is an external publication or a contributor referencing a not-yet-registered edition (the ETSI/#329 shape — a contributor needed `etsi-en-304-223` and it was absent). A bump may be batched with the content that motivated it (the #329 + #273/#274 bundle is the worked example) or stand alone. There is no calendar sweep; the registry advances only when there is a reason and an owner.

### D11. ETSI EN 304 223 (#329) — register now in this ADR's form

ETSI EN 304 223 is the imminent 7th framework. #329 registers it; the persona-mapping proposals #273 (`personaModelProvider` → `Developers`) and #274 (`personaEndUser` → `End-users`) block on the registration. The #329 draft proposes an **interim** ISO-style free string and explicitly defers the durable convention to #318 (= this ADR). Rather than land ETSI on a free string and re-migrate it under #343, **this ADR sets ETSI's durable form**, and ETSI is the first worked instance of the D8/D9 controlled vocabulary.

#### D11a. Registry entry (verified metadata)

The framework keys in per D1/D2 with metadata verified against the ETSI EN 304 223 V2.1.1 PDF (cover page + document properties, ref REN/SAI-0022, edition 2025-12):

- **`id`:** `etsi-en-304-223` (concept id, the mapping key; added to the `framework.id` enum and `framework-mapping-patterns`, D7).
- **`version`:** `2.1.1` (V2.1.1).
- **`versionId`:** `etsi-en-304-223@2.1.1` (generated per D2a — concept id `@` normalized version).
- **`applicableTo`:** `[personas]` (narrow start per the #329 draft; widen to `controls`/`risks` only when those conventions exist).

#### D11b. Controlled vocabulary — the five stakeholder roles, version-pinned

ETSI EN 304 223 Table 4-1 defines a **closed set of five stakeholder roles**: `Developers`, `System Operators`, `Data Custodians`, `End-users`, `Affected entities`. This is the first instance of the D8/D9 controlled vocabulary, encoded as a closed, version-pinned `enum` (D7) — **not** a free string. The pinned values are:

`Developers@2.1.1`, `System Operators@2.1.1`, `Data Custodians@2.1.1`, `End-users@2.1.1`, `Affected entities@2.1.1`

So #274's mapping is `personaEndUser → [End-users@2.1.1]`, generated by the D4 tool, not hand-typed. Capitalization and hyphenation are exact, fixed by the enum (`End-users`, never `End Users` or `Endusers`) — the standardization the bare string could not deliver. Clause 4's note that "a single entity can hold multiple stakeholder roles" is expressed as multiple enum values in the mapping array, not by merging roles.

#### D11c. Supersession

ETSI ENs re-version (V2.1.1 → V2.2.1 → …). When the next edition publishes, D10's process applies: a tracking issue advances `version`, mints `etsi-en-304-223@2.2.1`, sets `supersedes: etsi-en-304-223@2.1.1`, adds the V2.2.1 enum, and runs the drift sweep over the ETSI persona mappings. Existing `@2.1.1` pins stay valid against the edition they were authored against.

#### D11d. Relationship to #329

#329 remains the execution vehicle (registry entry, schema enum + `framework-mapping-patterns`, style-guide section, sample persona mapping for #273/#274). What changes is the *form* #329 adopts: the controlled-vocabulary `enum` + `@2.1.1` version token defined here, **superseding the interim `{"type": "string"}` bare string the #329 draft proposed**. This is additive to #329's already-scoped work — the same five role names, now as a closed pinned enum — and it folds into the #343 migration so ETSI lands consistent with the other non-ID frameworks from the start. The #329 draft's forward-link to #318 is now satisfied by this D11.

## Alternatives Considered

- **Version only at the registry, latest-wins.** Hold version metadata solely on the framework registry entry and resolve every mapping against the current version. Rejected because it delivers version *representation* (a generated `versionId` for citation + `supersedes` lineage) but **not integrity**: the registry `versionId` is redundant exactly where OWASP's year-in-value already self-pins, and under latest-wins every mapping silently follows the registry forward, so the ATLAS-retire and EU-renumber cases re-resolve without notice. Per-value pinning (D3) breaks that coupling; the registry `version`/`versionId` (D1/D2) is retained as source-of-truth and citation, not as resolution.
- **Version-in-mapping-key (`mappings.mitre-atlas-v5`).** The literal reading of "version in the ID." Rejected as the primary collision: it grows the D5b *key* allowlist per version, forces the `framework.id` enum and consumer `propertyNames` to accept suffixed keys, and re-keys every content block on every re-version. Value-pinning (D3) delivers the same per-mapping version distinction with the version in the *value*, leaving keys/enum/`propertyNames` byte-stable (D7).
- **CoSAI structured form across all frameworks (`<framework>:<version>:<axis>:<sub>`), #318 Option C.** Rejected (D6): it invents a CoSAI dialect contributors must learn, diverges from every upstream's identifier practice, and loses spec-native readability. Per-value pinning achieves the integrity goal (version in the value) by riding a token on the otherwise spec-native ref, without the dialect.
- **Free string for the non-ID frameworks (`{"type": "string"}`).** A bare string for ISO 22989, the NIST Actor-Task axis, and the ETSI roles. Rejected (D8/D9/D11): a free string is not acceptable for a standard — `AI Partner (data supplier)`, `AI Partner (datasupplier)`, and `AI Part (Data supplier)` all validate and nothing flags the divergence. The roles/tasks are a closed published set, so a closed version-pinned inline `enum` (D7) rejects variants while staying spec-native and version-safe. This is the half of #153's instinct that is correct.
- **CoSAI structured form for non-ID frameworks only, #318 Option B.** Rejected: it would replace the roles with a CoSAI synthetic syntax (`iso22989:2022:ai_customer:ai_user`) for marginal machine-tractability gain while leaving native-ID frameworks divergent. The controlled-vocabulary `enum` (D8/D9/D11) achieves machine-tractability and rejects variants *using the spec's own role names*, without inventing a dialect.
- **Slug-plus-registry encoding for the controlled vocabulary.** Hold each non-ID framework's per-version vocabulary as a list in `frameworks.yaml` (alongside `version`/`priorVersions`) and have the drift validator join mapping values against it, with the schema entry left a coarse `pattern`. Rejected as the encoding (D7 commits inline `enum` instead): the known sets are small and stable, an inline `enum` is enforced directly by `check-jsonschema` with no extra validator, and self-containment in the schema is simplest to drift-test. The slug-plus-registry encoding is worth revisiting only if a vocabulary outgrows an inline enum or must live next to the version metadata for the D10 process to update in one place.
- **Single flat (version-blind) controlled vocabulary per non-ID framework.** A closed `enum` that is *not* keyed per edition. Rejected: it standardizes spelling but reintroduces the silent-re-resolution failure D3 closes — if a later ISO/ETSI edition adds, drops, or renames a role, a flat enum either silently admits the new edition's value against an old pin or forces a destructive edit. Per-framework-version enums (D7) keep each edition's set distinct, matching the D3 pinning property.
- **ETSI on a free string now, migrate under #343 later (the #329 draft's interim proposal).** Rejected (D11): landing ETSI as `{"type": "string"}` means registering it on exactly the form this ADR rejects for the other non-ID frameworks, then re-migrating its handful of values a second time. Since the five Table 4-1 roles are known and closed today, ETSI is registered directly in the controlled-vocabulary form and #329 adopts that form — one landing, not two.
- **Tooling-normalized canonical form, prose-at-YAML (#318 Option D).** Rejected: it introduces a prose↔canonical registry and a normalization layer (a new governed content surface) to solve a machine-tractability problem the spec-native-plus-token forms plus the generator (D4) already address.
- **Per-mapping version annotation as a separate field (`[{id, version}]`).** Rejected: it breaks the bare-string-array value shape every consumer and D5b regex assumes, and splits the version from the identifier it pins (a reader/validator must join two fields). Pinning the version *into* the value string (D3) keeps the bare-string-array shape and keeps identifier and version atomically together — the same property OWASP's `LLM02:2025` already has.
- **Stored `mappingId` field on each mapping.** A persisted id alongside each value, giving the generator tool a handle stable across value edits. Rejected as the default (D4b commits a derived, non-stored `mappingId`): a stored field changes the value shape in all four consumer schemas, whereas the derived form composes the handle from `cosai-id` + framework + value and leaves the consumer schemas untouched. Introducing a stored field is a future amendment if stable-across-edits addressing is ever required, not an implementation choice for #343.
- **Authoring pinned values by hand.** Rejected: hand-authored pins are the source of the wrong-version/typo errors the integrity mechanism exists to prevent, and they carry real authorial overhead. Pinned values are generated (D4) from a structured selection by `framework-mapping-maintainer.py`, exactly as `versionId`, templates, and tables are generated; the contributor supplies a selection, not a pinned string.
- **NIST axis: re-purpose `nist-ai-rmf` to free-form (Option B) or structured discriminator (Option D).** Rejected (D9): Option B breaks existing control Subcategory mappings; Option D forces a migration of every `GOVERN-X.Y` and leans on the structured form D6 rejected. A separate `id` (Option C) preserves the Subcategory surface untouched.
- **`versionId` as a build-only / consumer-derived value, not materialized in `frameworks.yaml`.** Rejected (D2b): re-deriving it in each consumer invites drift, gives the supersession fields (D2c) no stable on-disk target, and gives the pinned-value token (D3) no materialized vocabulary to validate against.

## Consequences

**Positive**

- **Mapping drift becomes expressible and version-checkable.** Per-value pinning (D3) records the edition each mapping was authored against, so re-versioning no longer silently re-points an existing mapping (semantic shift and renumber are prevented from happening silently because old and new values are distinct), and the drift validator (D5 Tier 1) can check every pin against the known version set and the controlled vocabulary. The retire/renumber *catalog-absence* detection (a retired ATLAS technique, a renumbered EU article) is enabled but not delivered at adoption — it is the Tier-2 enhancement, which pinning makes possible once per-version catalogs exist.
- **No key collision and no consumer-schema change.** The mapping keys, the `framework.id` enum, `propertyNames`, the four consumer schemas, and ADR-016 D3 are byte-stable (D7). Because `mappingId` is derived, not stored (D4b), mapping values stay bare-string arrays. The version dimension grows only in the D5b *value* patterns and the content values — and that migration is performed by tooling, not contributors.
- **The authoring cost is a structured selection, not a hand-typed token.** `framework-mapping-maintainer.py` (D4) generates pinned values from a structured selection, validating `--version` against the registry at authoring time (the earliest catch point) and purity-checking the result (D4c) — the same generate-not-author contract as `versionId`, templates, and tables.
- **The hybrid direction is honored.** A dedicated registry `version` field (authored source-of-truth, D1) plus a generated `versionId` (D2) gives "version visible and generated, not hand-picked," and the pinned value carries that same version token into every mapping (D3).
- **Multi-version coexistence is first-class.** Two editions of the same framework can be mapped simultaneously during a transition window (`AML.T0043@5.0.1` and `@6.0.0`); a `priorVersions` membership marks the older pin as sanctioned so Tier 1 does not flag it (M3).
- **Mappings stay interpretable.** Spec-native base forms plus a single trailing version token (D6) keep a mapping readable to a practitioner who knows the source framework, avoiding a CoSAI dialect.
- **Non-ID values are standardized, not free strings.** The controlled vocabulary (D8/D9/D11), encoded as an inline `enum` enforced by `check-jsonschema` (D7), rejects spelling and structural variants (`AI Part (Data supplier)`) that a free string admits silently. Standardization comes from the spec's own role/task table, version-pinned, so it does not invent a CoSAI dialect — the half of #153's instinct that was right.
- **STRIDE is guarded by its enum.** STRIDE carries no version token (`version: null`) but is integrity-safe because its six-value closed enum is frozen by the schema (D6); membership is enforced and the set changes only via a D10-gated schema edit. STRIDE is pinned by enum, not by token.
- **Version changes are a defined operation scaled to blast radius.** D10 turns a framework re-issuance into a reviewed operation: a zero-affected-pins bump is a lightweight registry edit, and the full issue-gated checklist (registry update → vocab refresh → drift sweep → per-mapping decision → re-doc) triggers only when drift flags affected mappings (M2). Nothing advances via an unreviewed in-place edit; "retain an old pin" is a first-class, `priorVersions`-sanctioned outcome.
- **ETSI lands once, in the durable form.** D11 registers ETSI EN 304 223 (#329) directly as the controlled-vocabulary + version-token shape, so #273/#274 unblock without a later re-migration off an interim free string. ETSI is the worked proof of the D8/D9 mechanism.
- **The trio (plus ETSI) resolves coherently in one ADR.** #303 (D1/D2/D3/D5), #318 (D6/D8/D9/D11), #319 (D9), the version-change process (D10), and #329 ETSI (D11) land together with one collision analysis (D7).

**Negative**

- **The immediate guarantee is narrower than full dangling-detection.** What ships (D5 Tier 1) is version-known plus controlled-vocabulary membership; detecting that a referenced technique was retired or an article renumbered (Tier 2) needs per-version catalogs CoSAI does not yet hold. Pinning is the prerequisite that makes Tier 2 possible later, but adopters must not read this ADR as catching the ATLAS-retire or EU-renumber cases at adoption.
- **Integrity and standardization have a real, named schema + content cost.** The four versioned ID-bearing value patterns gain a version dimension, the three non-ID-bearing entries change from `{"type": "string"}` to closed version-pinned inline enums, the `framework.id` enum gains two keys (`nist-ai-rmf-actor-tasks`, `etsi-en-304-223`), and the content migrates: 96 `mappings.<framework>:` blocks comprising 146 individual values (risks 100, controls 40, personas 6), across 46 entities on disk today in risks/controls/personas (D7). EU AI Act is *prospective* — zero content today, entering via #164 — so it adopts the pinned form at first authoring rather than being migrated. The cost is paid in the value layer by tooling, but it is real debt handed to #343 / #319 / #329.
- **The controlled vocabulary must be maintained per edition.** Each non-ID framework's closed inline enum is sourced by hand from the spec's role/task table and must be refreshed on every version bump (D10b step 2). A missing or stale enum entry blocks a legitimate mapping until the vocabulary is updated — the standardization trades "anything validates" for "only the published set validates, and the published set must be kept current."
- **The full version-change path adds maintainer overhead.** When a bump affects existing pins, D10 gates it behind a tracking issue with a checklist — a reviewed migration, not a one-line `version` edit. The lightweight path (M2) keeps zero-affected bumps cheap, but for high-cadence frameworks (ATLAS quarterly) a bump that *does* affect pins recurs as real work.
- **A new generator tool, a purity validator, and a drift validator.** `framework-mapping-maintainer.py` (D4), its purity check (D4c), and the Tier 1 drift-detection validator (D5) are three new tooling contracts in the generated-artifact class. Tier 2 drift detection additionally needs per-version framework catalogs (a later enhancement, not a prerequisite).
- **A two-token-style split (OWASP `:YYYY` vs `@<version>`).** D6 retains OWASP's `:` form while everything else uses `@`. Justified (OWASP's `:` is upstream-recognizable and already committed) but the style guide must teach both, and consumers parsing a pinned value to recover its version must select the delimiter per framework, never assume one character (M1).
- **Two NIST framework ids from one source spec** (D9) is unusual and will read oddly to a contributor expecting one id per document; the style guide must explain the axis split.
- **Pins go stale and need maintenance.** A mapping pinned to `nist-ai-rmf@1.0` does not auto-advance when the registry moves to `1.1`; that is the point (no silent re-resolution), but it means re-pinning (or sanctioning a retention via `priorVersions`) is a deliberate maintenance action the drift validator (D5 Tier 1) surfaces and a maintainer must action. The project trades silent-and-wrong for visible-and-stale-until-reviewed.

## Addendum 2026-06-10: mandatory-pin enforcement completes the D7/M1 block phase; validate-all.sh content-schema parity

**Status:** Draft (maintainer to flip to Accepted)

Authored 2026-06-10 alongside [#343](https://github.com/cosai-oasis/secure-ai-tooling/issues/343) (PR #377). Two enforcement gaps surfaced after the strict-schema flip landed; both are now closed. This addendum records the decisions; it does not change any D-numbered decision.

- **The mapping-purity validator now FAILS (no longer skips) an unpinned value on a versioned framework — the D7/M1 "block" phase.** During migration `validate_mapping_purity.py` (D4c) classified a versioned-framework value lacking a `@`/`:` token as `skip` (the pre-migration "warn" tolerance, mirroring the C2 linter flip). Migration is complete and the strict consumer schemas make pinning mandatory, so a bare versioned value (e.g. `GOVERN-6.2` without `@1.0`) is now invalid: the validator returns `fail` with an "unpinned, version token required" detail. This is the warn→block transition (D7/M1) for the purity validator specifically. **The drift validator (D5) is intentionally unchanged** — currency/supersession is its job; "is it pinned at all" is owned by purity plus the schema. The resulting asymmetry is deliberate: on the same unpinned value (`GOVERN-6.2` with no token) purity now returns `fail` while drift returns `skip` — an unpinned value carries no version token for drift's three-state taxonomy (D5a) to classify, so drift defers to purity and the schema rather than double-reporting the same defect; the `skip` is by design, not an oversight. STRIDE (unversioned, `version: null`) is unaffected: its tokenless values are pinned by enum, not by token (D6) — an in-enum value classifies `ok`/`current`, and only a legacy out-of-enum spelling classifies `skip`.
- **`validate-all.sh` now runs content `check-jsonschema` for the four consumer YAMLs (CI-parity).** The manual full-tree sweep previously invoked `check-jsonschema` only as `--check-metaschema` (validating the schema *files*), never validating the content YAMLs against the consumer schemas. So the mandatory-pin gate that pre-commit and CI enforce was absent from the sweep — a maintainer unpinning a value got a false all-clear locally. The sweep now validates each of `risks`/`controls`/`components`/`personas` against its schema (mirroring the per-file `schema: <X>.yaml` pre-commit hooks), so the three checks (`check-jsonschema`, mapping-purity, mapping-drift) agree on an unpinned value.
