| componentToolRegistry                             | Tool Registry and Discovery                  | The infrastructure that serves as the registration authority for tools available to AI agents: cataloging and indexing them, holding the authoritative metadata (tool descriptions, parameter schemas, capability declarations, publisher identity information, invocation methods, and org-specific use restrictions), and publishing authoritative pointers to the tool endpoints where invocations are made. This includes tool registries, MCP server manifest repositories, tool discovery endpoints, schema definition services, and namespace governance systems.<br><br> Registry scope intentionally covers BOTH selection metadata (which tool to invoke) AND invocation metadata (how to invoke it). Selection and invocation are tightly coupled in real tool-registry formats — OpenAPI specs, MCP manifests, and tool schemas bundle "what this does" with "how to call it." Org-specific policies that restrict invocation methods (e.g., masking out invocation paths deemed too risky for a given deployment context) are enforced at the registry layer because that's where they sit between the agent and the tool. Splitting selection and invocation across separate components would force a parallel "use restrictions" component without addressing the coupling.<br><br> Tool registries serve a fundamentally different function from tools themselves (External Tools and Services). Tools are the external APIs and services that agents invoke (the tool functionality); the registry is the catalog/metadata layer that informs which tool is selected and how the agent should invoke it. The threat models are distinct: attacks on tools target functionality (Insecure Integrated Component), while attacks on a registry target the metadata that directs tool selection (Tool Source Provenance, Tool Registry Tampering). A compromised registry can redirect an agent to the wrong tool — or to a corrupted invocation method on the right tool — even when the tools themselves remain secure.<br><br> Registries are decoupled from tools: the same tool may be referenced by multiple registries from different sector-focused authorities (compliance-flavored, performance-flavored, licensing-flavored), and registries are not required in all deployments — internal-only or air-gapped agent systems may operate without any public registry layer. Controls targeting registry integrity apply only where a registry is part of the deployment.<br><br> The registry's scope also extends to the *publication and registration* step through which extensible artifacts — plugins, connectors, skill packages, and MCP server bundles — are made available and added to the catalog before they are installed into the agentic system. | componentToolServer                               | Tool Server                                  | A **tool server** is the remote or out-of-process endpoint that hosts a set of capabilities (tools, callable resources, and prompt templates) and exposes them to an agent over a tooling protocol such as MCP, A2A, or a remote plugin interface ([Model Context Protocol Specification](https://modelcontextprotocol.io/specification)).<br><br> It owns the serving and brokering boundary where capabilities are advertised, discovered, and invoked. It is distinct from the External Tools and Services it fronts: rather than being the underlying service, the tool server stands between the agent and those services, authenticating callers, presenting downstream credentials, and delegating to downstream services on behalf of multiple users.<br><br> Its key distinction is deployment-dependent trust posture: an instance can range from a local subprocess sharing the host's user context to a multi-tenant remote service, so its trust boundary and isolation posture vary by deployment pattern — a zero-trust posture that verifies trust at each boundary rather than assuming it from position or prior connection ([NIST SP 800-207: Zero Trust Architecture](https://csrc.nist.gov/pubs/sp/800/207/final)) applies regardless of deployment form.<br><br> A tenant-isolation failure at a multi-tenant tool server is not hypothetical: in the 2025 Asana MCP incident ([Asana warns MCP AI feature exposed customer data to other orgs](https://www.bleepingcomputer.com/news/security/asana-warns-mcp-ai-feature-exposed-customer-data-to-other-orgs/)), a flaw in tenant-session handling let one organization's data reach other organizations' users, affecting approximately 1,000 customers. If input that ultimately reaches the agent's request path is attacker-influenced, the tool server can be driven to exercise that standing access on the attacker's behalf, disclosing data or taking actions the caller was never authorized to reach directly. The exposure sits in the width of the credential the tool server itself holds and presents on the caller's behalf.<br> Because callers reach this endpoint over a network, authentication at this boundary must be mutual rather than assumed: the connecting client and the server establish identity through mechanisms such as bearer tokens, OAuth, or mTLS before any capability is exposed, and weak, absent, or replayable authentication here lets an unauthorized client drive the hosted tools or lets a malicious server impersonate a trusted one. Capability exposure is also negotiated per connection rather than fixed in advance: the server determines, for each connecting client, which capabilities that client may invoke, and a server that over-advertises capabilities or mutates an advertised tool's behavior after the client has already approved it expands the agent's effective authority beyond what the operator intended. Because it fronts the External Tools and Services it serves, the tool server also inherits their deceptive-tool-description and least-privilege concerns, so capability advertisements arriving from downstream must be treated as untrusted input subject to validation, not accepted at face value.                                                                                                                                                                                                                                                                                                                                              | componentsTools          |
